### Step D — Form tampering self-renewal defense

**Action:** Via DevTools JS:
```javascript
const editForm = [...document.querySelectorAll('form')].find(f => f.action.toLowerCase().includes('edittraining'));
const inp = document.createElement('input');
inp.type = 'hidden';
inp.name = 'RenewsTrainingId';
inp.value = '33';  // self.Id
editForm.appendChild(inp);
// Then click Simpan Perubahan button (preserve CSRF)
```

**Expected per spec:** Reject dengan "Sertifikat tidak boleh renewal dirinya sendiri."

**Actual:** ✅ Tampering **REJECTED**. Error displayed: `"Tanggal renewal harus lebih besar dari tanggal sertifikat yang di-renew."` (P03 DAG message, NOT self-renewal-specific message).

**FINDING — Validator interaction in self-renewal scenario:**

Self-renewal (RenewsTrainingId == self.Id) triggers TWO validators in EditTraining POST:
1. **P03 DAG** (`src.Tanggal >= model.Tanggal`) → since src=self, src.Tanggal == model.Tanggal, strict `>=` rejects equal per D-10
2. **Self-renewal guard** (`model.RenewsTrainingId.Value == model.Id`)

Edit handler L469-477 `firstError` translation surfaces only the **first** ModelState error → DAG message wins.

**Functional impact:** ZERO — tampering still rejected (D-07 defense-in-depth goal met). User UX impact: error message slightly less precise but actionable.

**Optional follow-up (not blocking):** Re-order validators (self-renewal first) di Plan 01 Task 3 Mod 2 untuk surface dedicated message. Acceptable as-is for v19.0 ship.

## Findings & Notes

### FINDING #1: Validator order in self-renewal scenario (non-blocking)

Edit POST has 4 validators in order: DAG-TR → DAG-AS → Self-renewal → P06. For self-renewal injection where FK=self.Id, DAG-TR fires first (src.Tanggal == model.Tanggal equality rejects). `firstError` translation surfaces DAG message, hiding dedicated self-renewal message.

**Impact:** Functional defense intact (tampering rejected). UX message slightly less precise.

**Recommendation:** Acceptable for v19.0. Future enhancement: re-order validators (self-renewal first) OR add custom check `if self-renewal { surface dedicated message AND skip DAG }`.

### FINDING #2: Worker re-selection after validation fail (pre-existing UX)

When AddTraining POST validation fails and returns View(model), Tom Select multi-worker field shows worker NOT pre-selected in re-render (hidden input value preserved but listbox UI loses visual selection). User must re-select worker.

**Impact:** UX papercut, NOT a Phase 326 regression — pre-existing pattern from prior phases. Out of scope this milestone.
