## Spacing Scale

The app uses Bootstrap's spacing utilities (`.25rem`/4px step). This phase reuses them; do not introduce custom px values. Mapping of Bootstrap classes used in this phase:

| Token | Value | Bootstrap class | Usage in this phase |
|-------|-------|-----------------|---------------------|
| xs | 4px | `g-1`, `gap-1`, `py-0` / inline `.25rem` | Gap inside `.img-drop` row between thumb/meta/button is `gap: .5rem`–`.55rem` → use `gap-2` (8px) on the flex row |
| sm | 8px | `mb-2`, `gap-2`, `px-2` | Spacing between an option's `input-group` and its image field; padding inside option image field |
| md | 16px | `mb-3`, `card-body` padding, `g-4` columns | Spacing below the question image field; below the last option field before divider |
| lg | 24px | `py-4` container, `mb-4` header | Existing page rhythm — unchanged |

Exceptions (carried from existing form + mockup, all multiples of 4 or Bootstrap-native):
- Thumbnail box: **46px × 46px** (matches mockup `.thumb`). This is an intentional fixed control size for a compact preview swatch, analogous to a 44px touch target. Allowed.
- Image-field container padding: `.5rem .65rem` (Bootstrap `p-2`-ish). Round to `px-2 py-2` (8px) when implementing.
- `input-group-text` correct-radio cell width `36px` (existing form L154) — unchanged.

## Color

The app inherits Bootstrap theme colors. Brand reference (from approved mockup): Pertamina blue `#0a4d8c` + green accent `#1a936f`. In this admin form the brand colors appear via Bootstrap's primary/success roles already present in `_Layout`; do NOT hardcode hex unless replicating the mockup's `.img-drop` states.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` / Bootstrap `bg-white` + `#eef1f5` page bg | Card body surfaces, form background |
| Secondary (30%) | `#f8f9fa` (`bg-light`) | Card headers, section boxes, empty image-field background (`#f6f9fc`) |
| Accent (10%) | Primary blue `#0a4d8c` (`btn-outline-primary`, `text-primary`, focus rings) | "Pilih" / "Ganti" file buttons; field labels emphasis; focus outline |
| Success (state) | Green `#1a936f` / Bootstrap `#198754` | Image-present state: `.img-drop.has-img` solid green border + green filename + `bi-check-circle-fill` |
| Destructive | Bootstrap `#dc3545` (`btn-outline-danger`, `text-danger`) | Remove-image (×) button; "Hapus gambar" checkbox label; top error alert (`alert-danger`) |

Accent reserved for (explicit): file-pick/replace buttons, field labels, focus rings, and the section title. NOT applied to plain body text or every interactive element. Green is reserved exclusively for the "image successfully attached" affordance. Red/danger is reserved exclusively for destructive (remove) and validation-error surfaces.

Image-field state colors (from mockup `.img-drop`):
- Empty: dashed border `#b6c2cf`, background `#f6f9fc`, gray placeholder icon `bi bi-image` on `#e2e8f0`.
- Has image: solid border `#1a936f` (green), background `#f1faf6`, green filename text.

## Component Inventory (this phase)

Reusable from existing app — do not rebuild:
- Top alerts: `alert-success` / `alert-danger` dismissible (L28–35) — image validation errors surface here (D-08).
- `input-group input-group-sm` option rows A–D (L150–163) — image field is appended directly UNDER each row (D-01 inline).
- Preview modal `#previewModal` + `loadPreview()` (L214–235) — renders `_PreviewQuestion` partial; image `<img>` added inside it (D-07).
- AntiForgeryToken, hidden `packageId` / `questionId`, JS prefill via `loadEditForm()`.

New components to build (Razor + vanilla JS), styled per mockup:
1. **Image field block** (`.img-drop`) — one per question + one per option A–D. Three visual states:
   - **Empty:** dashed gray box · `bi bi-image` placeholder thumb (46px) · meta text "Belum ada gambar" · `Pilih` button (`btn btn-outline-primary btn-sm`).
   - **Selected (client, pre-submit):** solid green box · live FileReader thumbnail · meta `<b>filename</b> · size` · red `×` clear button (`btn btn-outline-danger btn-sm`).
   - **Has saved image (edit prefill):** solid green box · `<img src=ImagePath>` thumbnail · meta filename · `Ganti` button + `×` remove button + "Hapus gambar" checkbox.
2. **Alt-text input** — `<input type=text class="form-control form-control-sm">` placeholder `Teks alternatif (opsional)`, shown inside/under the image field once an image is present or being added.
3. **Image `<img>` in `_PreviewQuestion`** — question image above option list; per-option image inside each `list-group-item`. `img-fluid`, `loading="lazy"`, `alt="@ImageAlt"`, max-height cap (see below).

Image-field anatomy (flex row, per mockup): `[thumbnail 46px] [meta text, flex-grow] [action button(s)]`, `gap-2`, single line that wraps gracefully on narrow column.

Preview render sizing: question image `img-fluid` capped `max-height: 240px`, option images `img-fluid` capped `max-height: 120px`, both `rounded border`. (Default — mockup shows admin-form thumbnails only; participant sizing is Phase 354.)

## Interaction Contract

| Trigger | Behavior |
|---------|----------|
| Admin picks a file (`<input type=file>`) | JS FileReader renders instant 46px thumbnail in that field; field flips empty→has-img (green); meta shows filename + KB/MB; `×` clear button appears (D-07 client preview). No server round-trip. |
| Admin clicks `×` (clear, pre-submit) | Resets that file input; field flips back to empty state. |
| Admin clicks "Ganti" (edit mode, saved image present) | Opens file picker; on pick, new thumbnail replaces old; submitting replaces file on disk (D-04/D-05). |
| Admin checks "Hapus gambar" (edit mode) | Marks image for deletion on submit (IMG-06). |
| Conflict: "Hapus gambar" checked AND new file picked | New file wins — image replaced, old file deleted, checkbox ignored (D-05). UI need not block; backend resolves. Optional nicety: when a new file is picked, uncheck + disable the "Hapus" checkbox to mirror the resolution (planner discretion). |
| Submit with non-image / >5MB file | Server `ValidateImageFile` rejects → redirect back with `TempData["Error"]` top alert + form repopulated (D-08). NOT inline per-field. |
| Edit form load (`loadEditForm`) | `EditQuestion` GET JSON now carries `imagePath`+`imageAlt` for question + each option (D-06); JS prefills thumbnails (`<img src>`) and alt inputs; `<input type=file>` stays empty (browser security, D-03). |
| Preview (`loadPreview` / after save) | `_PreviewQuestion` renders saved images via `<img>` (D-07, RND-04). |

Form attribute change: `#questionForm` MUST gain `enctype="multipart/form-data"` (D-02) — without it no file uploads. This is a contract requirement, not a style choice.

## Accessibility

- Every rendered image gets `alt="@ImageAlt"` (empty string `alt=""` when alt is null → decorative, valid).
- Icon-only clear/remove buttons get `aria-label` (see copy table).
- File input associated with a visible label; "Pilih"/"Ganti" implemented as `<label class="btn ...">` wrapping the file input (mockup pattern) so it is keyboard-focusable.
- "Hapus gambar" checkbox has an explicit `<label for>`.
- Image-present state must not rely on color alone: pair green border with the `bi-check-circle-fill` icon + bold filename (already in mockup).
