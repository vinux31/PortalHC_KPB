# Phase 361 — UI Design Contract: Bypass UI (B)

> Visual and interaction contract for the Tab2 "Bypass Tahun" redesign of `/ProtonData/Override`. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Stack reality:** Server-rendered ASP.NET Core MVC Razor + Bootstrap 5.3 + Bootstrap Icons + vanilla JS. NO SPA, NO Tailwind, NO shadcn (not a React/Next/Vite project — shadcn gate not applicable). This contract therefore reuses the established Bootstrap design system already live in `Views/ProtonData/Override.cshtml` and `Views/Admin/ManageAssessment.cshtml`. **Do not introduce new tokens** — the value of this phase is visual consistency with Tab1, which must remain behaviorally unchanged (PBYP-08, CONTEXT D-01).

## Interaction Contract

State, navigation, and async behavior the executor must implement. All consume the FINAL 6 endpoints at `ProtonDataController.cs:1499-1682` (+ one sanctioned D-18 select extension).

### Tabs & deep-link
- Page becomes 2 Bootstrap tabs (`nav nav-tabs` pattern from `ManageAssessment.cshtml:68`). Tab1 = existing Override markup wrapped in a `tab-pane` **without behavioral change** (D-01, PBYP-08).
- Default load = Tab1 (`?tab=deliverable` or no param). Tab2 data **lazy-loads** on first activation (D-07).
- Manual tab switch updates query param via `history.replaceState` (`?tab=bypass` / `?tab=deliverable`) so refresh/bookmark restores the tab (D-08).
- Deep-link `?tab=bypass&pending={id}` → activate Tab2 → load pending panel → **auto-open "Lihat & Konfirmasi" modal** for `{id}` (1-click from bell, D-05). Stale `{id}` → info toast + normal Tab2 (D-06).

### Wizard (modal multi-step, D-01)
- Single Bootstrap modal, 3 steps in JS state (mirrors `overrideModal` shell). Step indicator `1-2-3` across top (D-02).
- Linear navigation: `[Lanjut]` disabled until current step valid; `[Kembali]` allowed (D-02).
- Step 1 Tujuan: select target track (validate `|Δtahun|≤1` client-side from `BypassDetail.sourceTahun`; backend re-validates).
- Step 2 Closure mode: render 4 mode cards; enable only modes in `BypassDetail.eligibleModes`; ineligible = card `opacity-50` + muted reason text (D-10).
- Step 3 Detail: TargetUnit = **required** cascading Bagian → Unit dropdown from OrganizationUnits (D-11, reuse Tab1 cascade JS); Coach dropdown **optional** default `Pertahankan coach saat ini` (D-12); Alasan textarea required; then recap + warnings (D-03).
- Coach dropdown data source is an open planner decision (CONTEXT risiko) — do NOT hardcode; resolve via ViewBag from `Override()` GET or a small endpoint. UI contract only requires a `<select>` with a "Pertahankan coach saat ini" default option.

### Pending panel
- ALWAYS rendered (D-16). Heading shows live count `Menunggu Konfirmasi (N)`.
- Status label is descriptive (D-17): DB `Menunggu` → badge `Menunggu Exam` (yellow); DB `Siap` → badge `Siap Dikonfirmasi` (green). DB values unchanged.
- Row buttons: `[Lihat & Konfirmasi]` only when Status="Siap" (D-13); `[Batal]` always.

### "Lihat & Konfirmasi" modal (D-18)
- Populated from extended `BypassPendingList` select: worker identity, S→T, unit, coach, alasan, exam result + score, exam completion date.
- **Sanctioned backend change (D-18):** extend `BypassPendingList` select (`ProtonDataController.cs:1534`) to add: exam score, exam completion date, `Reason`, target-coach name. Query-select only — NO migration, NO change to existing field contract. Planner must verify real `AssessmentSession` column names (`Score`/`IsPassed`/`CompletedAt`) before extending.
- Buttons: `[Konfirmasi Pindah]` (success) / `[Batal]` (danger).

### Async / anti-double-click (D-21)
- Every POST (`BypassSave`/`BypassConfirm`/`BypassCancelPending`) disables its button + shows inline `spinner-border spinner-border-sm` while in-flight (matches `btnSaveOverride` pattern `Override.cshtml:410-413`).
- On success: close modal, toast (green), refresh worker table + pending panel (D-04, D-20).
- On backend reject (stale state, D-11 360): red toast with backend `message` + auto-refresh pending panel, modal closes (D-20).
