# Phase 367 — UI Design Contract

> Visual and interaction contract for the Delete Records Cascade Overhaul. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **CRITICAL CONSTRAINT:** This is an EXISTING ASP.NET Core MVC + Razor + Bootstrap 5.3 (offline) app. This contract LOCKS REUSE of established Portal HC KPB conventions. It introduces ZERO new tokens, frameworks, CSS files, or component libraries. Every value below is already live in the codebase — the contract's job is to pin which existing pattern each new surface adopts so the executor has no design ambiguity.

## Interaction Contract

The behavioral spine — this is the load-bearing part of the spec for this phase (honest-UI is the core requirement).

### IC-1 — Delete trigger opens preview, never deletes directly (S2 → S1)

- The per-row "Hapus" button does NOT post a delete. It triggers `GET DeletePreview(type, id)` whose response (a partial) is injected into the modal body, then the modal is shown.
- Loading uses `hx-target` into the modal's `#cascade-preview-body` + `hx-indicator` on the spinner, OR vanilla `bootstrap.Modal(...).show()` after HTMX swap. Reuse the existing `deleteAssessmentModal` orchestration shape from `_AssessmentGroupsTab.cshtml` (server-rendered action URLs via `@Url.Action`, `modal-dialog-scrollable`, `aria-live` loading region).
- `type` distinguishes `training` vs `assessment` (manual + online share the generic per-session endpoint per L-07). The button passes the row's `Id` and `Type`.

### IC-2 — Confirm submits cascade delete with honest result (S1 → S3)

- "Hapus Semua" posts to the refactored endpoint (`DeleteTraining` or generic per-session `DeleteManualAssessment`) including antiforgery token (existing pattern: `__RequestVerificationToken` in `hx-vals`/body) + the set of mirror-candidate IDs the admin left checked.
- **Success path:** endpoint returns `HX-Trigger: recordDeleted`. Two consumers fire:
  1. The existing hidden re-fetch div (`hx-trigger="recordDeleted from:body"`, `_TrainingRecordsTab.cshtml:178-183`) re-swaps the tab preserving filters.
  2. The re-rendered partial shows the **success** flash (S3) and the badge counts (S4) recompute to match the now-shorter list.
- **Failure path:** endpoint returns `HX-Trigger: recordDeleteFailed` with a payload message, HTTP status reflecting failure. The partial renders the **error** flash (red) with `{pesan}`. The list is NOT silently re-shuffled to look successful. This kills the false-success bug (#1) and silent-block bug (#2).
- The modal hides on both outcomes (success and handled error); the outcome is communicated by the in-partial flash, not by the modal staying open.

### IC-3 — Online rows become deletable (S2)

- Build on the Phase 371 `@if(row.IsOnline)` seam. Online rows currently render only a "Lihat hasil" eye button when `CanViewResult`. This phase ADDS the `btn-outline-danger` trash button to online rows.
- Online rows remain otherwise read-only: NO Edit button for online (only View + Delete). Manual + training rows keep Edit + Delete.
- Visual parity: the online delete button is identical in size/color/icon to the manual/training delete buttons (`btn-sm btn-outline-danger`, `bi bi-trash`), grouped in the same `d-flex gap-1`.

### IC-4 — Badge ↔ list consistency (S4)

- Per-type counts shown in the top worker row (`CompletionDisplayText`, `bg-info text-dark` badge) MUST equal the number of rows actually displayed per type in the expanded panel (D-01: RECOMPUTE = baris tampil).
- Acceptance (spec §3.3, option-neutral): the badge number and the list contents must not contradict each other under the displayed label. After a cascade delete + re-fetch, the recomputed badge reflects the reduced list.
- This is a data/formula fix in `WorkerDataService`; the visual contract is only "badge keeps existing `bg-info text-dark` style, number matches rows."

### IC-5 — Accessibility (reuse existing affordances)

- Modal: `role="alertdialog"`, `aria-labelledby` on title, `aria-live="polite"` on the loading/preview region (matches `deleteAssessmentModal`).
- Delete buttons keep descriptive `aria-label` (not just icon).
- Mirror-candidate checkboxes are real `<input type="checkbox">` with associated `<label>` (keyboard + screen-reader operable).
- Color is never the sole signal: success/error flash each carry an icon + text prefix; badge statuses carry text labels, not color alone.
