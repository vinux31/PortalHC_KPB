### 5. 397 WR-01 — unlink cross-batch correctness (service, CRUD) — MED

**Actual code** `Services/InjectAssessmentService.cs:736-791` (`UnlinkInjectGroupAsync`):
```csharp

{

        .Where(s => s.LinkedGroupId == injectGroupId && s.IsManualEntry)   // ⚠ ALL manual-entry sessions in the group
        .ToListAsync();
    ...
    // 4. Kasus B revert stiker ... remaining online sessions in same LinkedGroupId

        .Where(s => s.LinkedGroupId == injectGroupId && !injectIds.Contains(s.Id))   // ⚠ touches sibling batches
        .ToListAsync();

```
**The cross-batch problem (D-04):** in **Kasus A (adopt)** an inject batch JOINS an existing group
(`ResolveLinkContextAsync` :634-637 returns `rep.LinkedGroupId`). Two SEPARATE inject batches can both adopt the
same `LinkedGroupId`. Selecting *all* `IsManualEntry && LinkedGroupId==injectGroupId` therefore unlinks sessions
that belong to OTHER inject batches, and step-4's `remaining` revert can null online stickers still needed by
those siblings.

**Scoping reference — how link writes are group-scoped on the write side** `ResolveLinkContextAsync`
`Services/InjectAssessmentService.cs:615-650`:
```csharp
if (rep.LinkedGroupId.HasValue)                 // Kasus A: adopt existing group (shared by multiple batches!)
    return (rep.LinkedGroupId.Value, opposite, false, new List<AssessmentSession>(), rep);
// Kasus B: groupId = rep.Id; targetSessions = SEMUA sesi room target standalone (Title+Category+Schedule.Date)
```
And how the COMMIT path already tracks per-batch identity: it builds `successSessions`/`SuccessSessionIds`
(`:96,342-343`) and wires `LinkedSessionId` per worker (`:176-185`). The unlink must restrict its scope to the
specific inject sessions of the batch being undone — NOT every manual session that happens to share the group.

**Apply (D-04):** scope step-1 to the inject batch's own sessions only. Candidate keys to verify in code:
(a) the set of `SuccessSessionIds` returned at commit (if the caller passes a batch id), or
(b) the inject group key actually created by THIS batch (in Kasus B, `LinkedGroupId == rep.Id` IS batch-unique;
in Kasus A it is NOT). Bidirectional revert (D-12) stays, but limited to the unlinked batch's `LinkedSessionId`
pairs. **D-01:** confirm the real cross-batch reproduction (two adopt-batches sharing a group) before changing;
if the UI can only ever unlink a freshly-committed standalone (Kasus B) group, the Kasus A path may be
unreachable → narrow the guard rather than rewrite.

**Audit shape to preserve** `:794-806` — `ActionType = "LinkPrePostUndo"` (≤ MaxLength(50)), one AuditLog per
mutated session, in-tx `_context.AuditLogs.Add`. Skor/jawaban/status online NEVER touched (T-397-04).
