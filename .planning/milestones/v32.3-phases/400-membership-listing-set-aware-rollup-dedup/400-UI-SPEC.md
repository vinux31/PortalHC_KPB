## Component & Interaction Contract (phase-specific — the load-bearing section)

### The single UI change: Unit column text in `_RecordsTeamBody.cshtml:27`

**File:** `Views/CMP/_RecordsTeamBody.cshtml`
**Cell today (line 27):** `<td class="p-3">@(worker.Unit ?? "---")</td>`
**Model:** `@model List<HcPortal.Models.WorkerTrainingStatus>` — the cell binds to `worker.Unit` (a `string` on `WorkerTrainingStatus`).

**The cell markup does NOT change.** It stays a plain `<td class="p-3">` rendering `@(worker.Unit ?? "---")`. What changes is the **value** of `WorkerTrainingStatus.Unit`, computed server-side in `WorkerDataService.GetWorkersInSection` per the contextual rule (D-02). The view is value-driven; this keeps the markup byte-stable and the change purely in the data layer.

**Contextual value rule (D-02 — server computes, view renders as-is):**

| Render context | `WorkerTrainingStatus.Unit` value | Cell renders |
|----------------|-----------------------------------|--------------|
| **Filtered** by a unit (`unitFilter` non-empty) | the `unitFilter` value (the matched unit, e.g. `"X"`) | `X` |
| **Unfiltered** (whole Bagian / My-Records), worker has ≥1 active unit | all active units, primary-first comma-join: `string.Join(", ", units.OrderByDescending(uu => uu.IsPrimary).ThenBy(uu => uu.Unit))` | `X, Y` |
| Worker has **0 active** UserUnits (legacy / `Unit=null`) | `user.Unit` (may be null) → cell's `?? "---"` catches null | `---` |

Concise server expression (per CONTEXT D-02/D-05):
`Unit = !string.IsNullOrEmpty(unitFilter) ? unitFilter : (allActiveUnitsPrimaryFirstJoined ?? user.Unit)` — then the view's existing `?? "---"` is the final null guard.

**Ordering (D-04):** primary-first then alphabetical — `OrderByDescending(uu => uu.IsPrimary).ThenBy(uu => uu.Unit)`. This MUST match the Phase 399 `_PSign`/Excel ordering so the comma-join string reads identically across surfaces.

**Plain text only (D-02, consistent with 399 D-08):** the unfiltered "X, Y" value is a comma-joined **string**, NOT a set of badges/chips. Do NOT introduce the Phase 399 `bg-success` primary star/`Utama` badge here. Rationale: this records-team cell follows the `_PSign`/Excel plain-text convention (where primary-first ordering is the only primary signal), not the ManageWorkers/Profile badge convention. Badges here would diverge from the established cell idiom and is explicitly out of scope.

**`data-unit` attribute (line 18) — VERIFIED no client-side reader.** Grep across `**/*.{js,cshtml,ts}` confirms the only JS readers of `data-unit` are in `CoachCoacheeMapping.cshtml` and `CDP/HistoriProton.cshtml` (different views). In `RecordsTeam.cshtml` the unit filter is **server-side**: `#unitFilter` is sent as a query param to `RecordsTeamPartial` (`RecordsTeam.cshtml:403`), which returns this partial's HTML swapped into `#workerTableBody`; `data-unit` is never read for filtering. **Contract:** set `data-unit` to the same contextual value as the cell text (keep attribute ↔ visible text consistent), OR leave it bound to `@worker.Unit` (now contextual) — either is acceptable since it has no consumer (Claude's Discretion, CONTEXT). Recommended: leave `data-unit="@worker.Unit"` so it tracks the contextual value automatically with zero markup churn.

**Row count / pagination integrity (D-01):** the table renders **exactly 1 `<tr class="worker-row">` per worker** (no row duplication). `RecordsTeam.cshtml:408` counts `.worker-row` for the worker count, and `RecordsTeamPartial` paginates by `workerList.Count` — both stay accurate because the set-aware filter is a `.Any()` predicate (1 row/worker), not a JOIN fan-out. The UI MUST NOT emit per-unit duplicate rows (baris-per-unit was DITOLAK, D-01 / Deferred).

### No other UI surface changes

- **ManageWorkers (`Views/Admin/ManageWorkers.cshtml`):** display is NOT reworked (D-06) — it already shows all units via the Phase 399 `ViewBag.UserUnitsDict` badge cell. Only the backend `unitFilter` predicate (`WorkerController.cs:202-204` + `ExportWorkers:300-301`) becomes set-aware/active-only. **No view edit in this phase.**
- **Excel export (records team + worker export):** plain-text, primary-first comma-join — already governed by Phase 399 `399-UI-SPEC.md §B`. Phase 400 only ensures the set-aware filter feeds it the right rows; no new export formatting contract.
- **CMP analytics / renewal / Team View list (no `unitFilter`):** untouched — set-aware predicate does not fire without a unit filter, so 0 visual drift (D1=b verified by-construction, SC#3).
