<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser fetch -> RiwayatPercobaan GET | Client requests a worker's riwayat by sessionId (server-side RBAC re-checks) |
| Server partial HTML -> modal innerHTML | Server-rendered (@-encoded) HTML injected into the DOM |
| data-worker-name -> modal title | Worker name (user content) crosses into the DOM title |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-406-10 | Tampering (DOM XSS) | Modal title set from data-worker-name | mitigate | Title set via `.textContent` (NOT innerHTML). Worker name never parsed as HTML. (Pitfall 5.) |
| T-406-11 | Tampering (stored XSS) | Partial HTML dropped into #riwayatBody via innerHTML | mitigate | Body HTML comes from the SERVER PartialView where every user field is Razor `@`-encoded (Plan 01). innerHTML of already-encoded server HTML is safe; no raw client string assembly. e2e "xss" test proves inertness. |
| T-406-12 | Information disclosure | fetch reaching the riwayat of any session | accept (mitigated upstream) | Endpoint carries `[Authorize(Admin,HC)]` (Plan 01); page itself is Admin/HC-only. Per-worker ownership is a Phase 407 worker-endpoint concern. |
| T-406-13 | Denial of service | appUrl missing -> 404 on Dev sub-path (broken modal) | mitigate | fetch uses `appUrl('/Admin/RiwayatPercobaan?...')` (Lesson 385 PXF-01). Acceptance greps for appUrl, not a bare path. |
</threat_model>
