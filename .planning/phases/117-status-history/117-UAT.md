### 1. Submit Evidence Creates History Entry

result: pass

### 2. Approve Deliverable Creates Separate Entries Per Role
expected: Have a SrSpv approve the deliverable, then a Section Head (SH) approve it, then HC review it. Each approval should create a separate history row with the respective role — "SrSpv Approved", "SH Approved", "HC Reviewed" (or similar status strings). Check the database for 3 distinct rows with different ActorRole values.
result: issue
reported: "Sr Spv dan SH sudah approve tapi di database tidak update, cuman Coach yang masuk database"
severity: blocker

### 3. Reject Deliverable Records Reason
expected: Have a reviewer reject a deliverable with a rejection reason (e.g., "Evidence not sufficient"). Check the DeliverableStatusHistory table — a new row should exist with Status "Rejected" and the RejectionReason field containing the verbatim reason text.
result: pass

### 4. Re-submit After Rejection Creates Re-submitted Entry
expected: After a deliverable has been rejected (Test 3), re-submit evidence for the same deliverable. Check the DeliverableStatusHistory table — a new "Re-submitted" entry should appear. The original "Rejected" entry (with its reason) should still be present and unmodified in the history.
result: pass

### 5. Rejection Reason Preserved in History After Re-submit

result: issue
reported: "di website page detail deliverable status reject hilang, seharusnya tidak hilang"
