**FIX-01 — Reactivate cascade (around line 3432, CoachCoacheeMappingReactivate action):**

Find the code that reactivates ALL inactive ProtonTrackAssignments for the coachee. Change it to:
- Only reactivate assignments where `DeactivatedAt` is not null AND `DeactivatedAt` is within 5 seconds of the mapping's `EndDate` (the timestamp when deactivation happened). This correlates assignments to the specific mapping deactivation event.
- On reactivate, clear `DeactivatedAt` back to null.
- Also update the Deactivate cascade (around line 3373) to set `DeactivatedAt = DateTime.UtcNow` when deactivating assignments.

Pattern for reactivate filter:
```csharp
var deactivationTime = mapping.EndDate; // when mapping was deactivated
var assignments = await _context.ProtonTrackAssignments
    .Where(a => a.CoacheeId == mapping.CoacheeId
        && !a.IsActive
        && a.DeactivatedAt != null
        && EF.Functions.DateDiffSecond(a.DeactivatedAt.Value, deactivationTime.Value) >= -5
        && EF.Functions.DateDiffSecond(a.DeactivatedAt.Value, deactivationTime.Value) <= 5)
    .ToListAsync();
```

For each matched assignment: set `IsActive = true`, `DeactivatedAt = null`.

**FIX-02 — Assign idempotency (around line 3147, CoachCoacheeMappingAssign action):**

Before the loop that creates new ProtonTrackAssignments, for each track being assigned:
1. Check if an inactive ProtonTrackAssignment already exists for that coachee+track combo.
2. If found: reactivate it (`IsActive = true`, `DeactivatedAt = null`) instead of creating a new one. Its existing ProtonDeliverableProgress rows are already linked via FK and will automatically become visible again.
3. If not found: create new assignment as before (existing logic).

Do NOT create new ProtonDeliverableProgress rows for reactivated assignments — they already have them.
