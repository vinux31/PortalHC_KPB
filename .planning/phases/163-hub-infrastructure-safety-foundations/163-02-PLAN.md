1. **AkhiriUjian** (line 2237):
   The challenge: GradeFromSavedAnswers reads related rows and sets properties on the session object in memory — it cannot be a single ExecuteUpdateAsync. Use re-fetch-and-verify pattern:

   a. Keep the initial isInProgress check (fast-fail for obvious non-InProgress)
   b. After `await GradeFromSavedAnswers(session);`, BEFORE `await _context.SaveChangesAsync();`:
      - Detach the session entity from the change tracker: `_context.Entry(session).State = EntityState.Detached;`
      - Use ExecuteUpdateAsync with WHERE guard to atomically set Status/Score/CompletedAt/IsPassed/Progress ONLY IF the session is still InProgress:
        ```csharp
        var rowsAffected = await _context.AssessmentSessions
            .Where(s => s.Id == id
                     && s.StartedAt != null
                     && s.CompletedAt == null
                     && s.Score == null
                     && s.Status != "Cancelled"
                     && s.Status != "Abandoned"
                     && s.Status != "Completed")
            .ExecuteUpdateAsync(s => s
                .SetProperty(r => r.Status, "Completed")
                .SetProperty(r => r.CompletedAt, DateTime.UtcNow)
                .SetProperty(r => r.Score, session.Score)
                .SetProperty(r => r.IsPassed, session.IsPassed)
                .SetProperty(r => r.Progress, 100)
            );
        ```
      - If `rowsAffected == 0`: race lost — redirect with info TempData "Sesi sudah selesai atau dibatalkan." (no error, silent skip per locked decision)

   c. Remove the old `await _context.SaveChangesAsync();` call — replaced by ExecuteUpdateAsync
   d. The audit log and cache removal should only run when rowsAffected > 0

2. **ResetAssessment** (line 2139):
   Reset is a multi-step operation (archive + delete responses + reset fields). The race risk is two HC users clicking Reset simultaneously.

   a. Keep the existing manual status check as fast-fail
   b. After all the preparation work (archive attempt, delete responses, delete package responses, reassign packages), BEFORE the final SaveChangesAsync that resets the session fields:
      - Instead of setting properties on the tracked entity, use ExecuteUpdateAsync with a status guard:
        ```csharp
        var rowsAffected = await _context.AssessmentSessions
            .Where(s => s.Id == id
                     && s.Status != "Cancelled") // Cancelled is the only non-resettable status
            .ExecuteUpdateAsync(s => s
                .SetProperty(r => r.Status, "Open")
                .SetProperty(r => r.Score, (int?)null)
                .SetProperty(r => r.IsPassed, (bool?)null)
                .SetProperty(r => r.Progress, 0)
                .SetProperty(r => r.StartedAt, (DateTime?)null)
                .SetProperty(r => r.CompletedAt, (DateTime?)null)
            );
        ```
      - If `rowsAffected == 0`: redirect with TempData error
      - The archive + delete operations before this are safe to run even in a race — they're idempotent (deleting already-deleted responses is a no-op, adding a duplicate archive is harmless)
   c. Remove the old property-setting + SaveChangesAsync for the session reset — replaced by ExecuteUpdateAsync
   d. Keep SaveChangesAsync for the archive/delete operations that happen before the reset (those are separate entity operations)
