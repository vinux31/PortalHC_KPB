<task type="auto">
  <name>Task 2: Audit and fix CSRF protection on all POST actions</name>
  <files>Controllers/NotificationController.cs</files>
  <action>
Audit every [HttpPost] action across all controllers for [ValidateAntiForgeryToken]:

1. Known issue: NotificationController has class-level [IgnoreAntiforgeryToken] (line 9). This disables CSRF protection on MarkAsRead, MarkAllAsRead, and Dismiss POST endpoints. These are AJAX endpoints called from JavaScript.

   Decision needed: NotificationController POST endpoints are called via fetch() from JS. Two options:
   a) Remove [IgnoreAntiforgeryToken], add [ValidateAntiForgeryToken] to each POST, and ensure the JS sends the antiforgery token (extract from cookie or hidden field).
   b) Keep [IgnoreAntiforgeryToken] since these are low-risk actions (mark-as-read, dismiss) that only affect the authenticated user's own notifications (userId from claims, not from request body). The attacker gains nothing from CSRF here — marking someone's notifications as read is not harmful.

   Choose option (a) — add proper CSRF tokens. Even though the risk is low, consistency matters and the fix is simple:
   - Remove [IgnoreAntiforgeryToken] from class
   - Add [ValidateAntiForgeryToken] to each POST action
   - In the JS that calls these endpoints (likely in _Layout.cshtml or a notification script), add the antiforgery token header. The standard ASP.NET Core pattern: read from `input[name="__RequestVerificationToken"]` or use `@inject Microsoft.AspNetCore.Antiforgery.IAntiforgery` to get the token, then send as `RequestVerificationToken` header.

2. Verify all other controllers: grep for [HttpPost] without nearby [ValidateAntiForgeryToken]. From the initial scan, all other controllers appear to have [ValidateAntiForgeryToken] on every POST.

3. Check if any AJAX POST endpoints in other controllers also skip CSRF (search for fetch/XMLHttpRequest POST calls in JS files and verify corresponding controller actions have CSRF protection).

IMPORTANT: Search for the antiforgery token pattern used in existing AJAX calls in the codebase first (e.g., in StartExam.cshtml or other views with fetch POST) to follow the established pattern.
  </action>
  <verify>
    <automated>grep -n "HttpPost" Controllers/NotificationController.cs && grep -n "ValidateAntiForgeryToken\|IgnoreAntiforgeryToken" Controllers/NotificationController.cs</automated>
  </verify>
  <done>Every POST action across all controllers has [ValidateAntiForgeryToken]. NotificationController no longer uses [IgnoreAntiforgeryToken]. AJAX calls send the antiforgery token.</done>
</task>
