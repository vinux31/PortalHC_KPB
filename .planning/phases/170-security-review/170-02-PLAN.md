  <files>Views/Admin/AssessmentMonitoringDetail.cshtml, Views/Admin/ManageWorkers.cshtml, Views/Admin/ManagePackages.cshtml, Views/Admin/CoachCoacheeMapping.cshtml, Views/CDP/CoachingProton.cshtml</files>
  <action>
Audit all user-supplied data rendering for XSS, SQL injection, and open redirect:

**XSS — Html.Raw audit:**
From initial scan, Html.Raw is used in ~35 locations. Categorize each:

SAFE patterns (no fix needed):
- `@Html.Raw(Json.Serialize(...))` — server-generated JSON from model data, not user input
- `@Html.Raw(System.Text.Json.JsonSerializer.Serialize(...))` — same, server-generated
- `@Html.Raw(ViewBag.SavedAnswers ?? "{}")` — server-generated JSON
- `@Html.Raw(GetApprovalBadge(...))` — server-generated HTML from helper method with controlled enum values

NEEDS REVIEW:
- `@Html.Raw(session.UserFullName.Replace("'", "\\'"))` in AssessmentMonitoringDetail.cshtml line 306 — UserFullName comes from DB. The Replace only escapes single quotes but not `</script>` or other XSS vectors. Fix: use `System.Text.Encodings.Web.JavaScriptEncoder.Default.Encode()` or `@Json.Serialize()` instead.
- `@Html.Raw(Model.Title.Replace("'", "\\'"))` in AssessmentMonitoringDetail.cshtml line 608 — same issue with Title field.
- `@Html.Raw(confirmMsg)` in ManagePackages.cshtml line 141 — check if confirmMsg contains user data.
- `@Html.Raw(user.FullName.Replace("'", "\\'"))` in ManageWorkers.cshtml line 274 — same pattern, FullName from DB.
- `@Html.Raw(statusBadge)` in ImportWorkers.cshtml line 113 — check if statusBadge contains user-supplied data.

For each NEEDS REVIEW item: if the value comes from user-editable DB fields (FullName, Title), replace the `Html.Raw(x.Replace("'", "\\'"))` pattern with proper JS encoding. The safe pattern is: use `@Json.Serialize(value)` which handles all escaping, or move the value to a `data-` attribute and read via JS.

**SQL injection:**
Grep for `FromSqlRaw`, `ExecuteSqlRaw`, `FromSqlInterpolated`, string concatenation with SQL. ASP.NET Core + EF Core uses parameterized queries by default via LINQ, so this is likely clean. Confirm no raw SQL exists.

**Open redirect:**
Already fixed in AccountController (Url.IsLocalUrl check at line 121). Grep for any other `Redirect(` calls that take user-supplied URLs. Check for `returnUrl` parameters in other controllers.
  </action>

Audit all 8 IFormFile upload endpoints for security:

1. **AdminController.KkjUpload** (line 119) — uploads to wwwroot/uploads/kkj/{bagianId}/
2. **AdminController.CpdpUpload** (line 462) — uploads to wwwroot/uploads/cpdp/{bagianId}/
3. **AdminController.CreateAssessment** (line 1915, supportingDoc) — check where stored
4. **AdminController.ImportWorkers** (line 4343, excelFile) — Excel import
5. **AdminController.ImportQuestions** (line 5448, excelFile) — Excel import
6. **CDPController.UploadEvidence** (line 1066) — evidence file
7. **ProtonDataController.GuidanceUpload** (line 611) — guidance file
8. **ProtonDataController.GuidanceReplace** (line 679) — guidance replacement

For each endpoint, verify and fix:

a) **File type allowlist**: Check if there is an extension allowlist. If missing, add one:
   - KKJ/CPDP uploads: `.xlsx`, `.xls`, `.pdf`, `.doc`, `.docx`
   - Excel imports: `.xlsx`, `.xls` only
   - Evidence uploads: `.pdf`, `.jpg`, `.jpeg`, `.png`, `.doc`, `.docx`
   - Guidance uploads: `.pdf`, `.doc`, `.docx`, `.pptx`
   Read existing validation code first — some endpoints may already have this.

b) **File size limit**: Check if there is a max size check. If missing, add a reasonable limit (e.g., 10MB for documents, 5MB for images). Also check if there is a global limit in Program.cs (Kestrel MaxRequestBodySize).

c) **Path traversal**: Check if the filename is sanitized before use in Path.Combine. The safe pattern is `Path.GetFileName(file.FileName)` to strip directory components. Check if the upload path uses user-supplied values (like bagianId) — if bagianId is an int, path traversal via bagianId is not possible.

d) **Filename collision**: Check if files are stored with original names (risk: overwrite) or generated names (safe). If original names, consider using GUID-based names.

Read each upload action fully before making changes. Focus on adding missing validations without breaking existing functionality. If an endpoint already validates correctly, document it and move on.
  </action>

  </verify>
  <done>All 8 file upload endpoints validate file extension against an allowlist, enforce a size limit, and use Path.GetFileName to prevent path traversal. Findings and fixes documented in SUMMARY.</done>
