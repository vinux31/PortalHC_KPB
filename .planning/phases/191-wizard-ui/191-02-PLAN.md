### 9. Scripts section — Wizard Controller JS
Write a single IIFE `WizardController` that manages:

**State variables:**
- `var currentStep = 1;`
- `var visitedSteps = new Set([1]);`
- `var returnToConfirm = false;`
- `var passPercentageManuallySet = false;`

**Functions:**
- `goToStep(n)` — hide all `.step-panel` via `d-none`, show `#step-{n}`, call `updatePills()`, call `updateBackToConfirmButton()`, if n===4 call `populateSummary()`
- `updatePills()` — for each pill 1-4: if step < currentStep and in visitedSteps → `bg-success text-white` + `bi-check-circle-fill` (completed); if step === currentStep → `bg-primary text-white active` + `bi-circle-fill`; if in visitedSteps but not current → `text-primary border border-primary` (clickable); else → `text-muted border` disabled
- `validateStep(n)` — per-step validation per UI-SPEC: Step 1 checks Category + Title (+ Track if Proton); Step 2 checks at least 1 checkbox checked (in correct container based on Proton mode); Step 3 checks schedDateInput + schedTimeInput + DurationMinutes + Status. Returns bool. Adds `is-invalid` class to invalid fields, removes from valid ones.
- `populateSummary()` — reads form values and populates all `summary-*` elements. For peserta: detect Proton mode via `Category.value === 'Assessment Proton'`, collect checked boxes from correct container, show first 5 names comma-separated, if >5 show "...dan N lainnya" expand button. For ValidUntil: show value or "Tidak ada".
- `updateBackToConfirmButton()` — shows/hides `.btnBackToConfirm` buttons based on `returnToConfirm` flag

**Event listeners:**
- "Selanjutnya" buttons (`.btn-next`): `validateStep(currentStep)` → if valid, `goToStep(currentStep + 1)`
- "Sebelumnya" buttons (`.btn-prev`): `goToStep(currentStep - 1)`
- "Kembali ke Konfirmasi" buttons (`.btnBackToConfirm`): validate current step → if valid, `goToStep(4)`, set `returnToConfirm = false`
- "Edit" links (`.edit-from-confirm`): read `data-step`, call `goToStep(n)`, set `returnToConfirm = true`
- Pill clicks: check `visitedSteps.has(n)` before navigating
- Category change (`#Category`): detect if Proton → call `applyProtonMode(isProton)`, reset Step 2 selections, update `passPercentageManuallySet` flag and PassPercentage default from `data-pass-percentage`. If switching between Proton and non-Proton, clear all checkboxes in both containers.
- PassPercentage manual edit: set `passPercentageManuallySet = true`
- Token toggle (`#IsTokenRequired`): show/hide `#tokenSection`
- Form submit: run schedule combiner (`schedHidden.value = schedDateInput.value + 'T' + (schedTimeInput.value || '08:00') + ':00'`)

**Proton AJAX (preserve existing logic):**
- Track change triggers fetch to `GetEligibleCoachees` endpoint
- Response populates `#protonUserCheckboxContainer` with checkboxes
- `applyProtonMode(isProton)` function: if Proton → show `#protonEligibleSection`, hide normal `#userCheckboxContainer`; else → hide proton section, show normal container

**CRITICAL preservation requirements:**
- All `asp-for` bindings must match existing model properties exactly
- All field `name` attributes must produce the same POST body as today
- `UserIds` checkbox name must remain `UserIds` (list binding)
- Schedule hidden field combiner must run on form submit
- Success modal + `createdAssessmentData` JSON script must remain unchanged
- Section filter + text search JS for user list must work within Step 2
- `generateToken()` function for AccessToken must survive
- All existing ViewBag references must remain: `ViewBag.Categories`, `ViewBag.Users`, `ViewBag.Sections`, `ViewBag.ProtonTracks`, `ViewBag.CreatedAssessment`

**DO NOT:**
- Add new controller actions or AJAX endpoints
- Change any field names or asp-for bindings
- Remove the success modal or its initialization JS
- Use `form.checkValidity()` globally (per-step validation only)
- Disable hidden inputs (they must still POST)
