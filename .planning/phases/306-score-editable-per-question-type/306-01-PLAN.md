## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → AssessmentAdminController POST | Admin user submits form via authenticated session; HTML5 validation can be bypassed via DevTools |
| AssessmentAdminController → Database (PackageQuestions, AuditLogs) | Server-side enforcement layer; CK constraint `ScoreValue > 0` exists but no upper bound — app must enforce 1-100 |
| AuditLogService write → AuditLogs table | Best-effort write; failure must not break main flow (defensive try/catch) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-306-01 | Tampering | CreateQuestion + EditQuestion POST (server-side range bypass via DevTools removing HTML5 min/max) | mitigate | Server-side `if (scoreValue < 1 || scoreValue > 100)` check at start of POST action (Task 1 + Task 2). Reject with TempData["Error"] flash + redirect — NO silent coerce. |
| T-306-02 | Tampering | AuditLogService write fails (DB transient error) and main flow aborts | mitigate | All `_auditLog.LogAsync(...)` calls wrapped in try/catch with `_logger.LogWarning(auditEx, ...)` fallback. Main flow (SaveChangesAsync for the question entity) completes BEFORE audit log attempt — failure of audit does not roll back question save. (Task 1 + Task 2 both implement defensive wrap.) |
| T-306-03 | Repudiation | Audit log missing oldScore / newScore / affectedSessionsCount — admin denies who made score change | mitigate | EditQuestion-ScoreChange description string captures BOTH oldScore AND newScore AND affectedSessionsCount in single audit row. Plus actorName derived from `currentUser.NIP - FullName` for non-repudiation traceability. (Task 2.) |
| T-306-04 | Information Disclosure | affectedSessions count exposed in JSON response to admin | accept | Admin role already has full read access to `AssessmentSessions` table via existing `/Admin/ManageAssessment` page. Count of distinct sessions per question is a derived statistic, not new sensitive data exposure. RBAC at controller level (`[Authorize(Roles = "Admin")]` on AssessmentAdminController) gates access.
