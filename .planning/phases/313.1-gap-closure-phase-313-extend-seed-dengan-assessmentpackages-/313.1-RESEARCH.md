### Pattern 1: T-SQL `OUTPUT INSERTED INTO @table-var` for Identity Capture

**What:** Capture auto-generated IDs dari batch INSERT ke table-variable untuk downstream FK reference.

**When to use:** Saat INSERT batch row di parent table dan butuh ID-nya untuk INSERT child rows (hierarchical seed). Hindari cursor loop atau `SCOPE_IDENTITY()` yang hanya capture last row.

**Example (verified pattern dari Microsoft official docs):**
```sql
-- Source: docs.microsoft.com/sql/t-sql/queries/output-clause-transact-sql [CITED]
DECLARE @SessionIds TABLE (Id INT, Title NVARCHAR(450));

INSERT INTO AssessmentSessions (Title, Category, UserId, ...)
OUTPUT INSERTED.Id, INSERTED.Title INTO @SessionIds(Id, Title)
VALUES
    ('Phase 313 Timer Fixture Online ManualBeforeTime', 'Test Phase 313', @UserId, ...),
    ('Phase 313 Timer Fixture Online ManualAfterGrace', 'Test Phase 313', @UserId, ...),
    -- … 7 rows total
    ;

-- Sekarang @SessionIds berisi 7 (Id, Title) tuple — Id auto-generated
-- Cross-join untuk INSERT Packages:
DECLARE @PackageIds TABLE (Id INT, AssessmentSessionId INT);

INSERT INTO AssessmentPackages (AssessmentSessionId, PackageName, PackageNumber, CreatedAt)
OUTPUT INSERTED.Id, INSERTED.AssessmentSessionId INTO @PackageIds(Id, AssessmentSessionId)
SELECT
    s.Id,
    'Phase 313 Test Package',
    1,
    @Now
FROM @SessionIds s;

-- Hasil: 7 row di AssessmentPackages, 7 (Id, AssessmentSessionId) di @PackageIds
```

**Pitfall:** Output table-var hanya bisa **TABLE variable** atau temp table — bukan `INSERT … OUTPUT … INTO` ke real table dalam complex query (limitation .NET DB constraint chain). Pakai TABLE variable seperti contoh.

### Pattern 2: Cross-Join Template untuk Bulk Hierarchical INSERT

**What:** Generate Cartesian product `@ParentIds × @ChildTemplate` untuk INSERT child rows tanpa loop.

**When to use:** Saat setiap parent row butuh N child rows identik (Phase 313.1: setiap Package butuh 3 Questions identik; setiap Question butuh 4 Options identik).

**Example:**
```sql
DECLARE @QuestionTemplate TABLE ([Order] INT, QuestionText NVARCHAR(MAX), ScoreValue INT);
INSERT INTO @QuestionTemplate VALUES
    (1, N'Phase 313 Test Q1: Pilih jawaban yang benar.', 10),
    (2, N'Phase 313 Test Q2: Pilih jawaban yang benar.', 10),
    (3, N'Phase 313 Test Q3: Pilih jawaban yang benar.', 10);

DECLARE @QuestionIds TABLE (Id INT, AssessmentPackageId INT, [Order] INT);

INSERT INTO PackageQuestions (
    AssessmentPackageId, [Order], QuestionText, ScoreValue,
    QuestionType, MaxCharacters, Rubrik, ElemenTeknis
)
OUTPUT INSERTED.Id, INSERTED.AssessmentPackageId, INSERTED.[Order]
    INTO @QuestionIds(Id, AssessmentPackageId, [Order])
SELECT
    p.Id,
    t.[Order],
    t.QuestionText,
    t.ScoreValue,
    'MultipleChoice',  -- D-02
    0,                 -- MaxCharacters (MC type, ignore)
    NULL,              -- Rubrik
    NULL               -- ElemenTeknis
FROM @PackageIds p
CROSS JOIN @QuestionTemplate t;

-- 7 packages × 3 template rows = 21 questions
```

### Anti-Patterns to Avoid

- **NULL FK violation (Phase 309 anti-pattern):** Jangan hardcode integer untuk SessionId/PackageId/QuestionId — IDs auto-generated, capture via `OUTPUT INSERTED.Id INTO @table-var`. Validation guard via `THROW 50001` kalau subquery return NULL.
- **`SCOPE_IDENTITY()` fallback for batch INSERT:** Hanya return ID dari **last** row inserted dalam scope — fragile dan miss IDs untuk row 1..N-1. Pakai `OUTPUT INTO`.
- **Cursor loop (`DECLARE … CURSOR FOR …`):** Set-based T-SQL lebih cepat dan idiomatic; cursor = code smell.
- **Skip schema verification (Phase 313 lesson):** Wajib cek snapshot `IsRequired()` columns sebelum INSERT (parent CONTEXT.md: AccessToken='', BannerColor='bg-primary' wajib walaupun "boleh kosong" secara konseptual).
- **Selector substring match (Phase 312 WR-03 anti-pattern):** Jangan pakai `tr:has-text("Phase 313")` plain — bisa match substring di kolom lain. Pakai `escapeRegex` + regex anchor (existing pattern di `exam-taking.spec.ts:1613` sudah benar).
- **Hardcode session ID di test (e.g., `/CMP/StartExam/150`):** IDs generated; pakai title locator + parse dari URL.
- **Polling tight loop di Playwright untuk wait retry:** Pakai `waitForURL` dengan timeout sufficient (30s untuk SubmitExam karena retry 1+2+4=7s + processing).

## FK Topology and Cleanup Order

Verified dari `Migrations/ApplicationDbContextModelSnapshot.cs:2155-2339`:

```
AssessmentSession (parent)
    │
    ├── AssessmentPackage (Cascade) ─────► OnDelete(Cascade) [VERIFIED line 2333]
    │       │
    │       ├── PackageQuestion (Cascade) ─► OnDelete(Cascade) [VERIFIED line 2173]
    │       │       │
    │       │       └── PackageOption (Cascade) ─► OnDelete(Cascade) [VERIFIED line 2162]
    │       │
    │       └── (PackageUserResponse references via PackageQuestionId/PackageOptionId — OnDelete Restrict [VERIFIED line 2190, 2195])
    │
    ├── PackageUserResponse (Restrict) ──► OnDelete(Restrict) [VERIFIED line 2184]
    │       │ blocks Session DELETE jika ada response existing
    │
    └── UserPackageAssignment (Cascade) ──► OnDelete(Cascade) [VERIFIED line 2333]
```

**Critical implication:**
- **`AssessmentSession.Cascade` chain works** untuk Package/Question/Option/UserPackageAssignment — DELETE Session would cascade.
- **BUT `PackageUserResponse` has `OnDelete(Restrict)` to AssessmentSession AND PackageOption AND PackageQuestion** — kalau ada response existing, Session DELETE FAILS dengan FK violation.

**Cleanup order (D-08 + D-09 compliance):**

```sql
-- 1. PackageUserResponse pertama (Restrict on Session/Question/Option)
DELETE pur FROM PackageUserResponses pur
INNER JOIN AssessmentSessions s ON pur.AssessmentSessionId = s.Id
WHERE s.Title LIKE 'Phase 313 Timer Fixture%';

-- 2. UserPackageAssignment (Cascade on Session, but Restrict on Package — 
--    must DELETE before Package DELETE)
DELETE upa FROM UserPackageAssignments upa
INNER JOIN AssessmentSessions s ON upa.AssessmentSessionId = s.Id
WHERE s.Title LIKE 'Phase 313 Timer Fixture%';

-- 3. PackageOption (Cascade dari PackageQuestion — auto, but explicit safer)
DELETE po FROM PackageOptions po
INNER JOIN PackageQuestions pq ON po.PackageQuestionId = pq.Id
INNER JOIN AssessmentPackages ap ON pq.AssessmentPackageId = ap.Id
INNER JOIN AssessmentSessions s ON ap.AssessmentSessionId = s.Id
WHERE s.Title LIKE 'Phase 313 Timer Fixture%';

-- 4. PackageQuestion
DELETE pq FROM PackageQuestions pq
INNER JOIN AssessmentPackages ap ON pq.AssessmentPackageId = ap.Id
INNER JOIN AssessmentSessions s ON ap.AssessmentSessionId = s.Id
WHERE s.Title LIKE 'Phase 313 Timer Fixture%';

-- 5. AssessmentPackage
DELETE ap FROM AssessmentPackages ap
INNER JOIN AssessmentSessions s ON ap.AssessmentSessionId = s.Id
WHERE s.Title LIKE 'Phase 313 Timer Fixture%';

-- 6. AssessmentSession (existing line 57-58)
DELETE FROM AssessmentSessions WHERE Title LIKE 'Phase 313 Timer Fixture%';
```

**Note:** `OnDelete(Cascade)` di EF Core diset di Database FK constraint level (verified via snapshot). Manual SQL DELETE secara teknis bisa rely on cascade trigger SQL Server, tetapi **explicit DELETE chain di seed script lebih aman** karena:
1. Clear intent untuk reviewer.
2. Hindari surprise jika cascade rule nanti diubah.
3. Output count per step = debugging aid (`PRINT N'… deleted'`).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Identity capture batch INSERT | Manual `SELECT MAX(Id)` after INSERT | T-SQL `OUTPUT INSERTED.Id INTO @table-var` | Race condition (concurrent inserts), not transactional, fragile [CITED: Microsoft docs]. |
| Iterate session × 3 questions × 4 options | Cursor loop or `WHILE` | Cross-join `@SessionIds × @QuestionTemplate × @OptionTemplate` | Set-based = single statement, transactional, faster, idiomatic. |
| Wait for redirect chain in Playwright | `setTimeout` + `page.url()` polling | `await page.waitForURL(regex, { timeout })` | Built-in robust waiter; respects test isolation [CITED: playwright.dev/docs/api/class-page#page-wait-for-url]. |
| Banner text discovery | Hardcode in test (drift risk) | Extract verbatim from controller/view source (this research provides) | Source of truth = code; document literal ke RESEARCH.md untuk planner consume. |
| Login fixture | New auth flow per test | Existing `login(page, 'coachee')` di `tests/helpers/accounts.ts` | Already there. Re-use. |

**Key insight:** SQL Server T-SQL menyediakan native idiom (`OUTPUT INTO` + cross-join) untuk hierarchical INSERT yang lebih robust dari any ORM/loop alternative. Untuk Playwright, `waitForURL` + `getByText`/`locator('.alert-X')` cover semua banner assertion needs — no custom wait logic needed.

## Playwright Redirect Verification Pattern

Three approaches considered:

| Approach | Pros | Cons | Recommendation |
|----------|------|------|----------------|
| `page.waitForURL(regex)` | Built-in, robust, idiomatic Playwright | Doesn't capture intermediate hops | **Primary choice** |
| `page.on('response', …)` listener | Capture every response (including 302) | Setup harus sebelum action; complex untuk async chain | Use only kalau butuh assert specific status code |
| `page.waitForResponse(predicate)` | Wait for specific response | Match by URL/status; one-shot | Use kalau butuh capture specific 302 in middle of redirect chain |

**Best practice for Phase 313.1 tests:**
1. Click action: `await link.click()`
2. Wait for final URL (browser auto-follows redirects): `await page.waitForURL(/\/CMP\/Results\/\d+/, { timeout: 30_000 })`
3. Assert banner: `await expect(page.locator('.alert-danger')).toContainText(...)`

**For SubmitExam POST chain (D-22 — bump to 30s):**
- POST sent by JS retry handler (3 attempts × backoff 1+2+4 = 7s) at ExamSummary
- Backend Tier-1 logic returns 302 → /CMP/StartExam
- JS retry handler **detect** redirect URL match `/CMP/StartExam` and **stop retry chain** (line 204-207, CR-02 fix)
- Final state: page navigates to /CMP/StartExam OR custom JS banner fires

**For 313.2/5/6 (Tier-1) verification flow:**

## Common Pitfalls

### Pitfall 1: Cascade FK Block by `PackageUserResponse`

**What goes wrong:** Re-running seed setelah live UAT → some sessions punya `PackageUserResponse` rows (saved answers from previous test). DELETE Session → FK violation karena `OnDelete(Restrict)` to PackageQuestion/PackageOption.

**Why it happens:** EF Core configures `PackageUserResponse → AssessmentSession = Restrict`, `PackageUserResponse → PackageQuestion = Restrict`, `PackageUserResponse → PackageOption = Restrict` [VERIFIED snapshot].

**How to avoid:** Cleanup chain WAJIB include `DELETE PackageUserResponses WHERE AssessmentSessionId IN (…)` **first**, sebelum cascade chain.

**Warning signs:** SQL error code 547 ("DELETE statement conflicted with REFERENCE constraint").

### Pitfall 2: `OUTPUT INTO` to Real Table (Restriction)

**What goes wrong:** Try `OUTPUT INSERTED.Id INTO SomeRealTable(Id)` — fails kalau target real table punya FK constraint atau triggers.

**Why it happens:** SQL Server restriction — `OUTPUT INTO` target with FK constraints requires either MERGE or specific options [CITED: docs.microsoft.com/sql/t-sql/queries/output-clause-transact-sql].

**How to avoid:** Use **TABLE variable** (`DECLARE @T TABLE (…)`) — works without restriction. Phase 313.1 uses table-vars exclusively.

**Warning signs:** Error 332 "The target table … of the DML statement cannot have any enabled triggers if the statement contains an OUTPUT clause without INTO clause."

### Pitfall 3: Modal Blocks Click in Playwright

**What goes wrong:** Click "Resume" link → navigate StartExam → resume modal auto-shows (`#resumeConfirmModal` `data-bs-backdrop="static"`) — subsequent click on radio fails because modal blocks pointer events.

**Why it happens:** Modal `static` backdrop + `keyboard="false"` — must explicitly click `#resumeConfirmBtn` to dismiss.

**How to avoid:** Setelah `waitForURL('/StartExam/…')`, check + dismiss modal:
```typescript
const modalBtn = page.locator('#resumeConfirmBtn');
if (await modalBtn.isVisible({ timeout: 3000 }).catch(() => false)) {
  await modalBtn.click();
}
```
[VERIFIED existing pattern di `exam-taking.spec.ts:147-148`]

**Warning signs:** Click radio fails dengan error "element is not visible" atau timeout 30s.

### Pitfall 4: Confirm Dialog di Submit Button

**What goes wrong:** Klik "Kumpulkan Ujian" → `onclick="return confirm('Kumpulkan ujian?…')"` fires native browser dialog — Playwright by default tidak handle dan test stuck.

**Why it happens:** ExamSummary line 168 [VERIFIED]: `<button onclick="return confirm('Kumpulkan ujian? …')">`.

**How to avoid:** Register dialog handler **sebelum** click:
```typescript
page.once('dialog', dialog => dialog.accept());
await page.locator('button:has-text("Kumpulkan Ujian")').click();
```

**Warning signs:** Test timeout setelah click submit.

### Pitfall 5: Sibling Session Title Match

**What goes wrong:** `CMPController.StartExam` line 905-908 [VERIFIED] query packages via Title+Category+Schedule.Date join — kalau seed session Title duplicate (mis. 2 fixture punya Title sama by mistake), packages cross-pollinate.

**Why it happens:** "Sibling session" pattern — same Title+Category+Date = same exam batch, packages shared.

**How to avoid:** **Seed tetap pakai 7 unique Title** (existing): `Phase 313 Timer Fixture Online ManualBeforeTime`, `…Online ManualAfterGrace`, dst. Tidak ada cross-pollinasi karena setiap fixture Title unique. Tetap aman.

**Warning signs:** Tidak ada untuk 313.1; documented untuk awareness saja.

### Pitfall 6: SQL Server `IDENTITY_INSERT` not needed

**What goes wrong:** Developer mungkin coba enable `SET IDENTITY_INSERT AssessmentPackages ON` to control IDs.

**Why it happens:** Misconception bahwa harus control ID untuk FK consistency.

**How to avoid:** TIDAK PERLU. `OUTPUT INSERTED.Id INTO @table-var` capture auto-generated IDs deterministically. `IDENTITY_INSERT` bermanfaat hanya untuk migrate-from-other-system, bukan seed.

### Pitfall 7: T-SQL `[Order]` Keyword Escape

**What goes wrong:** `INSERT INTO PackageQuestions (… Order …)` → syntax error — `Order` is reserved keyword.

**Why it happens:** Column name `Order` di model maps to SQL column `[Order]` (escaped).

**How to avoid:** Always use square brackets `[Order]` di T-SQL seed.

**Warning signs:** SQL parse error pada line dengan `Order` column.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `SCOPE_IDENTITY()` per-row capture | `OUTPUT INSERTED.<col> INTO @table-var` | SQL Server 2005 (20+ years ago) | Standard now; SCOPE_IDENTITY only for single-row INSERT context |
| Cursor loop for hierarchical INSERT | Set-based CROSS JOIN with table-var | T-SQL evolution | Set-based always preferred — faster, transactional, idiomatic |
| Hardcode IDs in seed | Capture via OUTPUT + cross-join | Always (since SQL Server 2005) | Anti-pattern Phase 309 (NULL FK violation) directly addresses this |
| `data-testid` first | Role/text-based locators acceptable when no testid present | Playwright best practice 2023+ | Project legacy markup tidak punya data-testid for exam radio inputs — fallback ke `input.exam-radio[type="radio"]` + label text |
