<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser (HC) → Section CRUD endpoints | Form POST with packageId/sectionId/name — untrusted input |
| HC → CreateQuestion/EditQuestion sectionId | sectionId could reference a section in a different package (IDOR) |
| Section Name → rendered HTML | XSS via Nama Section in panel/grouped headers/dropdown |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-415-05 | Spoofing/Elevation | Section CRUD endpoints | mitigate | `[Authorize(Roles="Admin, HC")]` + `[ValidateAntiForgeryToken]` on all 4 endpoints + CreateQuestion/EditQuestion (match UpdateShuffleSettings). |
| T-415-06 | Tampering (CSRF) | All Section POST forms | mitigate | `@Html.AntiForgeryToken()` in every Section form + quick-button form. |
| T-415-07 | Cross-Site Scripting | Nama Section render (panel, group headers, dropdown) | mitigate | Razor `@` auto-encode; NO `@Html.Raw` in new Section markup. |
| T-415-08 | Information Disclosure / IDOR | sectionId on CreateQuestion/EditQuestion + EditSection/DeleteSection | mitigate | Validate section belongs to the package before assign; load section by id and confirm AssessmentPackageId match; reject/null on mismatch. |
| T-415-09 | Tampering | Duplicate SectionNumber | mitigate | AnyAsync pre-check + unique index (Plan 01) as defense-in-depth → DbUpdateException fallback. |
</threat_model>
