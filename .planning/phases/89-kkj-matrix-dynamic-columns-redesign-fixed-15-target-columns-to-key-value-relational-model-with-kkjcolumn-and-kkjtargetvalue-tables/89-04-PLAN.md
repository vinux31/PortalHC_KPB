# Plan 89-04: Assessment Flow — Update CMPController GetTargetLevel Callers

## Goal

Fix the two remaining `PositionTargetHelper.GetTargetLevel` callers in `CMPController.cs` (lines 1425 and 1556) to use the new async signature `await PositionTargetHelper.GetTargetLevelAsync(_context, kkjMatrixItemId, position)`. Verify the build is clean with 0 errors. Verify the full assessment flow works end-to-end with the new dynamic column schema.

## must_haves

- `CMPController.cs` line 1425 caller updated to `await PositionTargetHelper.GetTargetLevelAsync(_context, mapping.KkjMatrixItemId, assessmentUser?.Position)`
- `CMPController.cs` line 1556 caller updated to `await PositionTargetHelper.GetTargetLevelAsync(_context, mapping.KkjMatrixItemId, assessmentUser?.Position)`
- Both callers correctly pass `mapping.KkjMatrixItemId` (int) NOT `mapping.KkjMatrixItem` (object)
- `mapping.KkjMatrixItem` navigation object is no longer needed in the caller — review whether `.Include(m => m.KkjMatrixItem)` in the query is still required for other property access in those methods
- `dotnet build` succeeds with 0 errors across the entire solution
- No references to the old `PositionTargetHelper.GetTargetLevel` (sync, object-based) remain anywhere in the codebase

## Context

### Current state in CMPController.cs
Two callers use the old sync `GetTargetLevel`:

**Caller 1 (around line 1425)** — in an assessment result processing method:
```csharp
int targetLevel = PositionTargetHelper.GetTargetLevel(mapping.KkjMatrixItem!, assessmentUser?.Position);
```

**Caller 2 (around line 1556)** — in a second assessment result path:
```csharp
int targetLevel = PositionTargetHelper.GetTargetLevel(mapping.KkjMatrixItem!, assessmentUser?.Position);
```

Both are in `async Task<IActionResult>` methods that already use `await` elsewhere, so adding `await` here is straightforward.

### What `mapping.KkjMatrixItem` was used for
The old helper accepted the full `KkjMatrixItem` object to read Target_* properties via reflection. The new helper only needs `mapping.KkjMatrixItemId` (the FK int). After this change, `mapping.KkjMatrixItem` may no longer be needed in those methods.

However, `mapping.KkjMatrixItem` may also be used elsewhere in the same methods (e.g., for `mapping.LevelGranted` — wait, that's on mapping itself, not the item). Check if `mapping.KkjMatrixItem` is used for anything else (like logging, competency name display). If it's ONLY used in the old `GetTargetLevel` call, remove the `.Include(m => m.KkjMatrixItem)` from the query for a minor performance improvement. If it's used elsewhere, keep it.

### Also fix: Remove Kkj() related compilation errors
Plan 89-03 already updated `CMPController.Kkj()`. Verify that the Kkj() method changes from plan 89-03 are complete and do not introduce new issues. If plan 89-03 was only partially applied to CMPController, complete those changes here.
