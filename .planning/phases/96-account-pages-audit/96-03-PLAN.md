# Plan 96-03: Browser Verification - Authentication Check and Smoke Testing

**Created:** 2026-03-05
**Status:** Pending
**Owner:** gsd-executor (with user verification)

## Goal

Verify Profile and Settings pages work correctly in browser — test authentication redirect, Profile display with various user data scenarios, Edit Profile validation, and Change Password flow.

## Success Criteria (must_haves)

1. Unauthenticated users accessing /Account/Profile or /Account/Settings are redirected to Login page
2. Profile page displays correctly for users with complete profile data
3. Profile page displays correctly for users with incomplete profile data (empty fields show "-")
4. Avatar initials display correctly for multi-word names, single-word names, and empty names
5. Edit Profile validation works (phone numeric, email format)
6. Change Password works in local auth mode
7. Change Password form is hidden in AD mode (if AD mode enabled)
8. Success/error messages display and auto-dismiss after 5 seconds
9. All ACCT-01 through ACCT-04 requirements verified PASS

## Tasks

### Task 1: Browser Test - Authentication Redirect

**Action:** Verify unauthenticated users are redirected to Login page

**Steps:**
1. Open browser in private/incognito mode (not logged in)
2. Navigate directly to `https://localhost:7001/Account/Profile`
3. Verify redirect to `https://localhost:7001/Account/Login`
4. Navigate directly to `https://localhost:7001/Account/Settings`
5. Verify redirect to `https://localhost:7001/Account/Login`
6. Document results (PASS/FAIL with screenshots if applicable)

**Acceptance Criteria:**
- Profile redirects to Login when not authenticated
- Settings redirects to Login when not authenticated
- No raw exceptions or error pages shown

**Test Data:**
- Unauthenticated session (private browser window)

---

### Task 2: Browser Test - Profile Page Display (Complete Data)

**Action:** Verify Profile page displays correctly for user with complete profile data

**Steps:**
1. Login as user with complete profile data (e.g., admin user from Phase 87 seed data)
2. Navigate to `/Account/Profile`
3. Verify all fields display correctly:
   - Avatar initials (e.g., "AD" for "Admin User")
   - Nama Lengkap
   - NIP
   - Email
   - Telepon
   - Direktorat
   - Bagian
   - Unit
   - Jabatan
   - Role
4. Verify no "-" placeholders for fields with data
5. Verify page renders without errors
6. Document results

**Acceptance Criteria:**
- All profile fields display correct data
- Avatar initials show first 2 characters of first 2 words
- No null reference exceptions or missing data
- Page layout renders correctly

**Test Data:**
- Admin user from SeedTestData.cs (Phase 87)
- Or any user with complete profile from existing DB

---

### Task 3: Browser Test - Profile Page Display (Incomplete Data)

**Action:** Verify Profile page handles incomplete profile data gracefully

**Steps:**
1. Find or create user with incomplete profile (missing NIP, Phone, Position, etc.)
2. Login as that user
3. Navigate to `/Account/Profile`
4. Verify empty fields show "-" placeholder
5. Verify avatar initials handle edge cases:
   - Single-word name → first 2 characters
   - Empty name → "?"
6. Verify no null reference exceptions
7. Document results

**Acceptance Criteria:**
- Empty fields show "-" placeholder
- Avatar initials handle edge cases correctly
- No crashes or error pages
- Page renders with partial data

**Test Data:**
- User from existing DB with incomplete profile
- Or manually create test user with missing fields

---

### Task 4: Browser Test - Edit Profile Validation

**Action:** Verify Edit Profile form validation works correctly

**Steps:**
1. Login as any authenticated user
2. Navigate to `/Account/Settings`
3. Click "Edit Profile" tab if not already active
4. Test phone validation:
   - Enter "abc123" in phone field
   - Click "Simpan Perubahan"
   - Verify error: "Nomor telepon hanya boleh angka"
5. Test email validation (if editable):
   - Enter "invalid-email" in email field
   - Click "Simpan Perubahan"
   - Verify error: "Format email tidak valid"
6. Test successful edit:
   - Enter valid phone: "08123456789"
   - Click "Simpan Perubahan"
   - Verify success message appears and auto-dismisses after 5 seconds
7. Verify data persisted by checking Profile page
8. Document results

**Acceptance Criteria:**
- Phone numeric validation works
- Email format validation works
- Success message displays and auto-dismisses
- Data persists correctly

**Test Data:**
- Any authenticated user (e.g., worker from Phase 83)

---

### Task 5: Browser Test - Change Password (Local Auth Mode)

**Action:** Verify Change Password flow works correctly in local auth mode

**Steps:**
1. Ensure appsettings.json has `"UseActiveDirectory": false` (default)
2. Login as any user with local auth
3. Navigate to `/Account/Settings`
4. Click "Ubah Password" tab
5. Verify Change Password form is visible
6. Test password mismatch:
   - Enter wrong current password
   - Enter new password and confirm
   - Click "Ubah Password"
   - Verify error: "Password lama salah."
7. Test password too short:
   - Enter correct current password
   - Enter new password with 5 characters
   - Click "Ubah Password"
   - Verify error: "Password baru minimal 6 karakter."
8. Test successful change:
   - Enter correct current password
   - Enter new password with 6+ characters
   - Confirm new password
   - Click "Ubah Password"
   - Verify success message appears and auto-dismisses after 5 seconds
9. Verify user can login with new password
10. Document results

**Acceptance Criteria:**
- All password error messages are in natural Indonesian
- Success message displays and auto-dismisses
- Password change persists correctly
- User can login with new password

**Test Data:**
- Any user with local auth (e.g., worker from Phase 83)

---

### Task 6: Browser Test - Change Password (AD Mode)

**Action:** Verify Change Password form is hidden in AD mode

**Steps:**
1. Change appsettings.json to `"UseActiveDirectory": true`
2. Restart application
3. Login as any user (or skip login if AD auth not configured)
4. Navigate to `/Account/Settings`
5. Verify "Ubah Password" section shows info message instead of form:
   - "Password dikelola oleh Active Directory. Hubungi admin IT untuk mengubah password Anda."
6. Verify no Change Password form inputs are visible
7. Document results with screenshot
8. Revert appsettings.json to `"UseActiveDirectory": false` after test

**Acceptance Criteria:**
- Change Password form is hidden in AD mode
- Info message explains password is managed by AD
- No form inputs visible for password change

**Test Data:**
- Appsettings.json with UseActiveDirectory = true

---
