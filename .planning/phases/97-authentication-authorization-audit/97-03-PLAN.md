## Tasks

### Task 97-03-01: Fix critical security bugs from authorization matrix

**Action:** Fix any critical security gaps identified in plan 97-01 authorization matrix

**Common security bugs to fix:**

1. **Missing role gates:**
   - If Admin/HC-only action missing `[Authorize(Roles = "Admin, HC")]` → Add attribute
   - If Admin-only action missing `[Authorize(Roles = "Admin")]` → Add attribute
   - Example:
     ```csharp
     // Before (insecure - any authenticated user can access)
     [Authorize]
     public IActionResult DeleteWorker(string id)
     {
         ...
     }

     // After (secure - Admin only)
     [Authorize(Roles = "Admin")]
     public async Task<IActionResult> DeleteWorker(string id)
     {
         ...
     }
     ```

2. **Manual auth checks replaced with attributes:**
   - AccountController Profile/Settings use manual `User.Identity.IsAuthenticated` checks
   - Replace with `[Authorize]` attribute for consistency
   - Example:
     ```csharp
     // Before (manual check)
     public async Task<IActionResult> Profile()
     {
         if (User.Identity?.IsAuthenticated != true)
             return RedirectToAction("Login");
         ...
     }

     // After (attribute-based)
     [Authorize]
     public async Task<IActionResult> Profile()
     {
         ...
     }
     ```

3. **Cookie security gaps:**
   - If HttpOnly not set → Add `options.Cookie.HttpOnly = true`
   - If Secure not set (and SSL enabled) → Add `options.Cookie.SecurePolicy = CookieSecurePolicy.SameAsRequest`
   - If SameSite not set → Add `options.Cookie.SameSite = SameSiteMode.Lax`
   - Example in Program.cs:
     ```csharp
     builder.Services.ConfigureApplicationCookie(options =>
     {
         options.LoginPath = "/Account/Login";
         options.LogoutPath = "/Account/Logout";
         options.AccessDeniedPath = "/Account/AccessDenied";
         options.ExpireTimeSpan = TimeSpan.FromHours(8);
         options.SlidingExpiration = true;

         // Add security attributes
         options.Cookie.HttpOnly = true;
         options.Cookie.SecurePolicy = CookieSecurePolicy.SameAsRequest;
         options.Cookie.SameSite = SameSiteMode.Lax;
     });
     ```

**Fix process:**

1. For each critical bug from plan 97-01 matrix:
   - Read the affected file
   - Implement fix following examples above
   - Compile and test locally (dotnet build)
   - Commit with security-related message:
     ```
     fix(auth): add [Authorize(Roles="Admin")] to DeleteWorker action

     - Prevents unauthorized access to worker deletion
     - Critical security fix - any authenticated user could delete workers
     - Found in Phase 97 authorization matrix (plan 97-01)
     ```

**Verification:**
- All critical bugs fixed
- dotnet build succeeds
- Commits have clear security-related messages
- Fixes documented in phase summary

### Task 97-03-03: Test edge cases - multiple roles and session expiration

**Action:** Test edge cases not covered in plan 97-02 browser verification

**Edge case 1: Multiple roles resolution**

**Test:**
1. Assign user to both "Admin" and "HC" roles in database
2. Login as that user
3. Access Admin-only page (e.g., `/Admin/DeleteWorker`)
4. Access HC-accessible page (e.g., `/Admin/ManageWorkers`)

**Expected behavior:**
- User can access pages for ANY of their assigned roles
- `[Authorize(Roles = "Admin, HC")]` uses OR logic (user in Admin OR HC)
- Both Admin and HC features accessible

**Code check:**
- Verify ASP.NET Core Identity handles multiple roles correctly
- No custom role resolution logic needed (framework default)

**Edge case 2: Role change during login**

**Test:**
1. Login as Coachee
2. (In database) HC changes Coachee to Coach role
3. Navigate to Coach-accessible page (e.g., `/CDP/CoachingProton`)

**Expected behavior:**
- Coachee does NOT see Coach features (stale roles in session cookie)
- User must logout and login again to get new role
- Cookie contains role claims from login time, not real-time

**Code check:**
- Verify roles are embedded in session cookie at login time
- Verify no real-time role refresh on every request (correct design)

**Edge case 3: Session expiration**

**Test:**
1. Login as Admin
2. Wait for session to expire (8 hours per Program.cs, or set shorter timeout for testing)
3. Navigate to authenticated page (e.g., `/Admin/Index`)

**Expected behavior:**
- Redirect to `/Account/Login` with `returnUrl=/Admin/Index`
- User can login again and return to original page
- Session expiration handled gracefully (no errors)

**Code check:**
- Verify `options.ExpireTimeSpan = TimeSpan.FromHours(8)` in Program.cs
- Verify `options.SlidingExpiration = true` refreshes cookie on activity
- Verify `options.LoginPath = "/Account/Login"` handles expired sessions

**Documentation:**
- For each edge case tested, document:
  - Test performed
  - Expected behavior
  - Actual behavior
  - Any bugs found (or "no bugs found")

**Verification:**
- Edge case 1 tested (multiple roles)
- Edge case 2 tested (role change during session)
- Edge case 3 tested (session expiration)
- Results documented in phase summary
