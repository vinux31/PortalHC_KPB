5. Temuan Kritis & Risiko

5.1 Risiko Data Bentrok (Race Condition)

Race condition adalah situasi di mana dua pengguna melakukan aksi bersamaan, dan sistem tidak siap menanganinya — mengakibatkan data yang salah atau hilang.

# Masalah Penjelasan Sederhana Dampak Tingkat Risiko
1 Nomor sertifikat bentrok Jika 2 peserta submit ujian di waktu yang persis sama, sistem bisa membuat nomor sertifikat yang sama. Sudah ada mekanisme coba ulang (3 kali), tapi prosesnya lambat karena harus membaca semua sertifikat tahun ini. Nomor duplikat / error Menengah
2 File upload tertimpa Nama file menggunakan waktu upload (per detik). Jika 2 orang upload file dengan nama sama di detik yang sama, file pertama akan tertimpa oleh yang kedua — dan data hilang. Kehilangan file/dokumen Kritis
3 Sesi ujian ganda Jika peserta membuka ujian di 2 browser/tab bersamaan, sistem bisa membuat 2 sesi ujian. Database menangkap ini dan memberikan error 500 — bukan pesan yang ramah pengguna. Error 500 bagi peserta Menengah
4 Jawaban ujian tertimpa Jika peserta membuka ujian di 2 tab dan menjawab soal berbeda, jawaban dari tab yang lebih lambat bisa menimpa jawaban yang lebih cepat. Jawaban peserta hilang Menengah

5.2 Risiko Kecepatan Menurun

# Masalah Penjelasan Sederhana Dampak Tingkat Risiko
5 Koneksi database habis Sistem punya batas 100 koneksi database. Jika lebih dari 100 orang membutuhkan data di saat bersamaan, pengguna selanjutnya harus menunggu — atau mendapat error. Website tidak bisa diakses Kritis
6 Halaman Coaching Proton terlalu berat Setiap kali halaman dibuka, sistem membaca data dari 5 tabel bertingkat (seperti membuka 4 lapisan folder). Ini membuat halaman lambat dan database bekerja keras. Loading 5–30 detik Kritis
7 Notifikasi sertifikat satu per satu Saat HC membuka dashboard, sistem mengecek semua sertifikat expired dan mengirim notifikasi satu per satu. Jika ada 100 sertifikat × 10 penerima = 1.000 notifikasi individual. Dashboard HC loading sangat lama Kritis
8 Data tidak dioptimasi untuk baca saja Hampir semua pembacaan data menggunakan mode "siap edit" (membutuhkan memori lebih banyak) — padahal 90% pembacaan hanya untuk ditampilkan, tidak diedit. Memori server naik 30–50% lebih tinggi dari seharusnya Menengah
9 Import data tidak efisien Import pekerja/soal dari Excel memproses data satu per satu (bukan sekaligus). Import 500 soal membutuhkan ~1.500 permintaan database. Import lambat (menit vs detik) Menengah

5.3 Risiko Kehilangan Data

# Masalah Penjelasan Sederhana Dampak Tingkat Risiko
10 Tidak ada backup otomatis file File yang diupload (dokumen KKJ, evidence coaching, dll) disimpan di folder server tanpa mekanisme backup atau redundansi. Jika server bermasalah, file hilang Menengah
11 Operasi tidak atomik Beberapa proses yang melibatkan banyak langkah (contoh: update status + kirim notifikasi) tidak dijamin semua berhasil atau semua gagal. Jika gagal di tengah, data bisa setengah jadi. Data tidak konsisten Menengah

6. Rekomendasi Perbaikan

Prioritas 1 — Perbaikan Kritis (Harus Segera)

Perbaikan ini mencegah website down atau kehilangan data saat pengguna bertambah.

# Perbaikan Apa yang Dilakukan Tingkat Kesulitan Dampak
1 Naikkan batas koneksi database Ubah konfigurasi dari 100 menjadi 200 koneksi simultan. Perubahan kecil di 1 baris pengaturan. Mudah Cegah error massal saat peak
2 Amankan nama file upload Tambahkan kode unik (GUID) ke setiap nama file yang diupload, sehingga mustahil ada nama yang sama. Mudah Cegah file saling timpa
3 Optimalkan halaman Coaching Proton Ubah cara membaca data dari "ambil semua lalu saring" menjadi "ambil hanya yang diperlukan". Kurangi dari 18 query menjadi 5–7 query per halaman. Sedang Halaman 2–3× lebih cepat
4 Perbaiki notifikasi sertifikat Ubah pengiriman notifikasi dari satu per satu menjadi sekaligus (batch). 100 notifikasi = 1 perintah, bukan 100. Sedang Dashboard HC loading cepat

Prioritas 2 — Perbaikan Penting (Dalam Waktu Dekat)

Perbaikan ini meningkatkan kecepatan dan kestabilan secara signifikan.

# Perbaikan Apa yang Dilakukan Tingkat Kesulitan Dampak
5 Aktifkan mode "baca saja" untuk tampilan data Tambahkan instruksi ke database bahwa data yang hanya ditampilkan tidak perlu disiapkan untuk diedit. Menghemat 30–50% memori server. Mudah Kurangi beban memori
6 Tangani error sesi ujian ganda Jika peserta membuka ujian di 2 tab, tampilkan pesan yang jelas alih-alih error teknis. Mudah Cegah kebingungan peserta
7 Hapus query berlebih di Coaching Ada 3 permintaan database tambahan (dari perbaikan lama) yang sudah tidak diperlukan tapi masih berjalan. Mudah Kurangi 3 query per halaman
8 Optimalkan import Excel (batch processing) Ubah proses import dari "simpan satu per satu" menjadi "kumpulkan dulu, simpan sekaligus". Sedang Import 10× lebih cepat

Prioritas 3 — Peningkatan Jangka Panjang

# Perbaikan Apa yang Dilakukan Tingkat Kesulitan Dampak
9 Simpan data yang sering diakses di memori (cache) Data seperti daftar unit organisasi, kategori assessment — yang jarang berubah — disimpan sementara di memori server, tidak perlu dibaca ulang dari database. Sedang Kurangi beban database 20–30%
10 Pindahkan file ke cloud storage File yang diupload disimpan di layanan cloud (Azure Blob / AWS S3) dengan backup otomatis, bukan di folder server. Tinggi File aman, scalable, ada backup
11 Jamin operasi database yang lengkap (transaksi) Pastikan proses multi-langkah (update status + kirim notifikasi) berjalan semua atau gagal semua — tidak setengah-setengah. Sedang Data selalu konsisten

7. Roadmap Perbaikan

Minggu 1 — Perbaikan Darurat
Naikkan batas koneksi database + Amankan nama file upload
Perubahan konfigurasi kecil yang langsung mencegah 2 risiko kritis. Bisa dilakukan dalam 1 hari kerja.
Minggu 2–3 — Optimasi Performa
Optimalkan Coaching Proton + Notifikasi batch + Aktifkan mode baca saja
Perbaikan kode yang mengurangi beban database hingga 60%. Halaman Coaching dari 18 query menjadi 5–7.
Minggu 4 — Perbaikan UX & Error Handling
Tangani sesi ganda + Hapus query berlebih + Optimalkan import Excel
Perbaikan yang meningkatkan pengalaman pengguna dan kecepatan proses import data.
Bulan 2–3 — Peningkatan Arsitektur
Cache layer + Cloud storage + Transaksi database
Perubahan arsitektur yang mempersiapkan website untuk skala yang lebih besar (500+ pengguna).

Poin Penting untuk Pengambilan Keputusan

1. Website saat ini sudah berfungsi dengan baik untuk penggunaan sehari-hari dengan jumlah pengguna terbatas. Seluruh 200 fitur berjalan normal dan tidak ada bug fungsional yang ditemukan.

2. Risiko muncul saat pengguna bertambah banyak secara bersamaan — terutama saat pelaksanaan ujian assessment massal atau review coaching oleh banyak supervisor sekaligus.

3. Perbaikan Prioritas 1 (koneksi database + keamanan file) bisa dilakukan dalam 1 hari kerja dan langsung menghilangkan 2 dari 4 risiko kritis.

4. Perbaikan Prioritas 1 + 2 secara keseluruhan membutuhkan estimasi 3–4 minggu kerja, dan akan meningkatkan kapasitas website dari ~100 menjadi ~250 pengguna simultan.

Aspek yang Sudah Baik

Beberapa aspek website sudah mengikuti praktik terbaik:

  • Sistem login & otorisasi aman (role-based access control)
  • Proteksi submit ujian ganda sudah ada (status-guarded update)
  • Monitoring ujian real-time berfungsi baik (SignalR)
  • Random number generator sudah thread-safe
  • Cache untuk notifikasi sertifikat sudah diimplementasi
  • Audit log untuk aktivitas admin sudah aktif
