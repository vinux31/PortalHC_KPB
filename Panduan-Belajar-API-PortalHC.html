Apa itu API?

Konsep dasar yang wajib dipahami sebelum masuk ke teknis

Definisi Sederhana

API singkatan dari Application Programming Interface.

Dalam bahasa sehari-hari: API adalah "jembatan" atau "penghubung" antara dua sistem komputer agar bisa saling berbicara dan bertukar data.

🎯 Intinya: API memungkinkan satu aplikasi meminta data atau perintah ke aplikasi lain, tanpa perlu tahu "isi dapur" di dalamnya. Yang penting: minta → dapat jawaban.

📱 Contoh API di Kehidupan Nyata

Tanpa sadar, kamu sudah pakai API setiap hari:

  • Google Maps di Gojek — Gojek "minta" data peta dari Google Maps via API
  • Login dengan Google — Website minta Google konfirmasi identitas via API
  • Cuaca di HP — Aplikasi cuaca minta data dari server cuaca via API
  • Transfer bank via aplikasi — Aplikasi Anda minta server bank proses transfer via API
  • WhatsApp Web — Browser Anda minta pesan dari server WhatsApp via API

🎯 Kenapa API Itu Penting?

🚫 Tanpa API

  • Setiap aksi perlu reload halaman
  • Tidak bisa terintegrasi sistem lain
  • Tidak bisa dibuat aplikasi mobile
  • Data tidak bisa diakses dari luar
  • Sulit dibuat dashboard eksternal

✅ Dengan API

  • Data real-time tanpa reload
  • Integrasi ke SAP, Power BI, dll
  • Bisa dibuat aplikasi mobile
  • Data bisa diakses aman dari luar
  • Dashboard langsung dari data live

Cara Kerja API

Memahami alur request dan response secara detail

Alur Lengkap API

1

Client mengirim REQUEST

Browser atau aplikasi mengirim permintaan ke URL tertentu. Contoh: "Kasih saya data pegawai dengan ID 123"

2

Server menerima request

Server membaca permintaan, mengecek autentikasi (apakah user punya izin?), lalu meneruskan ke logika bisnis.

3

Server memproses data

Server mengambil data dari database, memproses kalau perlu, lalu menyiapkan jawaban dalam format JSON.

4

Server mengirim RESPONSE

Server mengirim balik data beserta kode status (200 = sukses, 404 = tidak ditemukan, 403 = tidak punya akses, dsb).

5

Client menerima & menampilkan

Browser/aplikasi menerima data JSON, lalu menampilkannya ke layar tanpa perlu reload halaman.

Jenis-jenis API

Tiga jenis yang relevan untuk PortalHC KPB

1. REST API — Yang Paling Umum

REST (Representational State Transfer) adalah jenis API yang paling banyak digunakan di dunia web. Prinsipnya sederhana: gunakan URL + HTTP Method untuk semua operasi data.

✅ Ini yang akan kita tambahkan ke PortalHC KPB. REST API cocok untuk semua fitur: ambil data pegawai, buat assessment, update training, dll.

Karakteristik REST API:

  • URL mewakili "sumber daya" (resource) — misal /api/pegawai
  • HTTP Method menentukan aksi (GET/POST/PUT/DELETE)
  • Data dikirim/diterima dalam format JSON
  • Stateless — setiap request berdiri sendiri, tidak bergantung request sebelumnya

2. WebSocket (SignalR) — Sudah Ada!

WebSocket adalah koneksi dua arah yang terus-menerus terbuka antara client dan server. Cocok untuk fitur real-time seperti chat, notifikasi langsung, atau monitoring live.

⭐ PortalHC KPB sudah menggunakan ini! Fitur monitoring ujian assessment menggunakan SignalR (implementasi WebSocket di ASP.NET). Peserta ujian mengirim jawaban real-time, dan HC bisa monitor secara live.

Perbedaan WebSocket vs REST:

REST API

  • Client minta → Server jawab → selesai
  • Koneksi tidak permanen
  • Cocok untuk CRUD biasa
  • Semua modul PortalHC (selain assessment)

WebSocket / SignalR

  • Koneksi terus terbuka dua arah
  • Server bisa kirim data kapan saja
  • Cocok untuk real-time
  • Assessment monitoring PortalHC ✅

3. Internal API (Controller Action)

Sebenarnya di PortalHC KPB saat ini sudah ada beberapa action yang mengembalikan data JSON — ini disebut internal API atau "quasi-API". Belum terstruktur sebagai REST API penuh, tapi sudah setengah jalan.

📌 Contoh yang sudah ada: SearchUsersApi, UnreadCount, GetOrganizationTree, RegenerateToken, dan beberapa lainnya sudah mengembalikan JSON.

Ini artinya tim developer sudah paham konsepnya — tinggal memperluas dan memformalkannya sebagai REST API yang terstruktur.

Potensi API per Modul

13 modul PortalHC KPB dan manfaat API di masing-masingnya

💡 Di bawah ini adalah analisis setiap modul PortalHC KPB: apa saja yang bisa dilakukan dengan API, dan integrasi apa yang bisa dibuka.

1. 🔐 Authentication (Login/Logout)

Kondisi saat ini: Login hanya bisa lewat form browser (tidak bisa dari aplikasi lain).

Dengan API:

  • Aplikasi mobile bisa login dengan email/password → dapat token (JWT)
  • Sistem lain bisa autentikasi ke PortalHC secara programatik
  • Single Sign-On (SSO) dengan sistem Pertamina lainnya

🎯 Use case nyata: Buat aplikasi mobile PortalHC — pegawai login dari HP, dapat notifikasi jadwal coaching, lihat progress CDP.

2. 👥 Pegawai (Worker Management)

Kondisi saat ini: Data pegawai hanya bisa diakses lewat halaman web.

Dengan API:

  • Power BI bisa pull data pegawai untuk dashboard visualisasi
  • Sinkronisasi otomatis dari sistem HR Pertamina (SAP, myHR, dsb)
  • Import/export data massal via sistem lain tanpa buka browser
  • Cek status aktif pegawai dari sistem lain

3. 📝 Assessment (Ujian Kompetensi)

Kondisi saat ini: Sudah punya beberapa endpoint JSON + SignalR real-time.

Dengan API REST penuh:

  • Jadwal assessment bisa diakses dari kalender eksternal (Google Calendar, Outlook)
  • Notifikasi otomatis ke WhatsApp/Teams saat assessment dijadwalkan
  • Hasil nilai bisa langsung masuk ke sistem pelaporan manajemen
  • Sertifikat bisa diunduh via API tanpa buka PortalHC

4. 🌱 CDP / Proton Track

Kondisi saat ini: Progress hanya bisa dilihat di PortalHC web.

Dengan API:

  • Dashboard progress CDP bisa tampil di Power BI untuk manajemen
  • Sinkronisasi deliverable dengan sistem Proton pusat Pertamina
  • Notifikasi otomatis ke coachee saat deliverable disetujui/ditolak
  • Laporan progress CDP per bagian untuk rapat manajemen

5. 📊 CMP (Competency Management Program)

Dengan API:

  • Data KKJ bisa diakses sistem lain untuk analisis gap kompetensi
  • Laporan alignment kompetensi masuk otomatis ke sistem pelaporan
  • Integrasi dengan LMS (Learning Management System) eksternal

6. 🤝 Coaching & Coach Mapping

Dengan API:

  • Jadwal coaching tampil di kalender Teams/Outlook otomatis
  • Coach bisa update log sesi dari HP (via mobile app)
  • Alert workload coach dikirim ke HC jika mendekati batas
  • Dashboard coaching per bagian untuk laporan manajemen

7. 🎓 Training Records

Dengan API:

  • Data training bisa disinkronkan dari sistem pelatihan eksternal
  • Laporan training masuk otomatis ke Power BI/Excel manajemen
  • Alert otomatis ke HC jika sertifikat hampir expired
  • Integrasi dengan sistem SAP HR Training

8. 🏢 Organisasi (Bagian/Unit)

Kondisi saat ini: Sudah ada GetOrganizationTree yang mengembalikan JSON.

Dengan API penuh:

  • Struktur org bisa ditarik sistem HR lain secara otomatis
  • Sinkronisasi perubahan struktur org dari sistem induk Pertamina

9. 📄 Dokumen (KKJ, CPDP)

Dengan API:

  • Dokumen KKJ bisa diakses dari sistem manajemen dokumen lain
  • Upload dokumen bisa dari aplikasi eksternal
  • Notifikasi otomatis saat dokumen baru diupload

10. 🔔 Notifikasi

Kondisi saat ini: Sudah ada UnreadCount sebagai API.

Dengan API penuh:

  • Notifikasi PortalHC bisa diteruskan ke WhatsApp, Teams, atau email
  • Sistem lain bisa kirim notifikasi ke pengguna PortalHC
  • Dashboard notifikasi di mobile app

11. 🏆 Sertifikat & Renewal

Dengan API:

  • Verifikasi sertifikat dari sistem eksternal (QR Code scan → API cek validitas)
  • Laporan sertifikat expired masuk ke email HC otomatis
  • Data sertifikat masuk sistem Pertamina pusat

12. ⚙️ Admin & Sistem

Dengan API:

  • Maintenance mode bisa diaktifkan dari sistem monitoring eksternal
  • Audit log bisa diintegrasikan ke SIEM (Security Information & Event Management)
  • Health check endpoint untuk monitoring uptime sistem

13. 📚 Proton Data (Kurikulum)

Dengan API:

  • Data kurikulum bisa disinkronkan dari sistem Proton pusat Pertamina
  • Update deliverable/silabus secara massal via import API
  • Laporan coverage kurikulum untuk manajemen

// Cara meminta data dari API (GET request)
async function ambilDaftarPegawai() {
  const response = await fetch('/api/pegawai', {
    method: 'GET',
    headers: {
      'Authorization': 'Bearer ' + userToken,
      'Content-Type': 'application/json'
    }
  });

  const hasil = await response.json();

  if (hasil.success) {
    // tampilkan data ke tabel HTML
    tampilkanTabel(hasil.data);
  } else {
    alert('Gagal: ' + hasil.message);
  }
}

API Controller di C# (ASP.NET Core)

Begini cara membuat endpoint API di PortalHC KPB menggunakan C#:

// File: Controllers/Api/PegawaiApiController.cs

[ApiController]
[Route("api/pegawai")]
[Authorize]  // harus login untuk akses
public class PegawaiApiController : ControllerBase
{
    private readonly ApplicationDbContext _db;

    public PegawaiApiController(ApplicationDbContext db)
    {
        _db = db;
    }

    // GET /api/pegawai — ambil semua pegawai aktif
    [HttpGet]
    public async Task<IActionResult> GetAll()
    {
        var pegawai = await _db.Users
            .Where(u => u.IsActive)
            .Select(u => new {
                u.Id,
                u.FullName,
                u.NIP,
                u.Position,
                u.Section,
                u.RoleLevel
            })
            .ToListAsync();

        return Ok(new { success = true, data = pegawai });
    }

    // GET /api/pegawai/{id} — detail 1 pegawai
    [HttpGet("{id}")]
    public async Task<IActionResult> GetById(string id)
    {
        var pegawai = await _db.Users.FindAsync(id);
        if (pegawai == null)
            return NotFound(new { success = false, message = "Pegawai tidak ditemukan" });

        return Ok(new { success = true, data = pegawai });
    }
}

Perbedaan MVC Controller vs API Controller

MVC Controller (yang sudah ada)

  • Return: View (halaman HTML)
  • Atribut: [Controller]
  • Base class: Controller
  • Dipakai browser biasa
  • Contoh: WorkerController

API Controller (yang akan ditambah)

  • Return: JSON data
  • Atribut: [ApiController]
  • Base class: ControllerBase
  • Dipakai aplikasi/sistem lain
  • Contoh: PegawaiApiController

✅ Keduanya bisa hidup berdampingan! Tidak perlu menghapus MVC controller yang sudah ada. Cukup tambah folder Controllers/Api/ yang baru.

API Key: Konsumen vs Penyedia

Memahami "kartu identitas digital" yang wajib dipakai saat integrasi antar sistem

Apa itu API Key?

API Key adalah "kartu identitas + tiket masuk" yang dikasih sebuah website ke kamu, supaya kamu boleh memanggil API mereka.

🎯 Analogi: Bayangin kamu mau masuk gym. Kamu daftar dulu, dapat kartu member dengan barcode unik. Setiap kali masuk, kamu tap kartu itu. Satpam tau: "Oh, ini Rino, member aktif, boleh masuk." Kalau kartu hilang/dicuri, orang lain bisa masuk pakai identitas kamu. API Key persis seperti kartu itu — string panjang acak.

Contoh bentuk API Key:

CONTOH_KEY_a7Bx9KpQmZ3nR8vT2wY6jL4dF1hG5cV

Kenapa Website Kasih API Key?

Server mereka butuh tau siapa yang manggil API, untuk:

  • Authentication — "Kamu siapa? Terdaftar nggak?"
  • Authorization — "Kamu boleh akses fitur apa aja?"
  • Rate limiting — "Kamu udah panggil 1000x hari ini, stop dulu."
  • Billing — "Pakai 50rb request bulan ini, tagihan sekian."
  • Audit log — "Jam 10:00, API Key ini panggil endpoint X."

Tanpa API Key, server tidak bisa bedain kamu vs hacker vs user lain.

Cara Pakai API Key

API Key dikirim bareng setiap request ke API mereka, biasanya di HTTP Header (bagian tak terlihat dari request, bukan di URL).

Contoh — memanggil API cuaca:

GET https://api.cuaca.com/v1/jakarta
Header:
  Authorization: Bearer CONTOH_KEY_a7Bx9KpQmZ3nR8vT2wY6jL4dF1hG5cV

Server cuaca cek: "Key valid → balas data." Kalau key salah → balas 401 Unauthorized.

⚠️ Jangan taruh API Key di URL query string (misal ?key=xxx). URL tercatat di log server, history browser, bahkan kadang terindex Google. Selalu di Header.

Dua Sisi API Key — Konsumen vs Penyedia

Ini yang sering bikin bingung. API Key ada di dua sisi:

📥 Kamu = Konsumen

  • Kamu daftar di website mereka → dapat API Key
  • Simpan key di server kamu (bukan di browser!)
  • Setiap call ke API mereka, attach key itu
  • Contoh: pakai SendGrid untuk email, Fonnte untuk WA, Midtrans untuk bayar

📤 Kamu = Penyedia

  • Kamu yang generate API Key dan kasih ke partner
  • Simpan key (hash) di database kamu
  • Verifikasi setiap request yang pakai key itu
  • Contoh: website teman kirim nilai ujian ke PortalHC

💡 Di kasus integrasi PortalHC ↔ website teman, keduanya terjadi: Teman kasih API Key ke kamu (supaya kamu bisa trigger "mulai ujian" di sistemnya). Kamu kasih API Key ke teman (supaya dia bisa kirim nilai balik ke kamu). Dua API Key berbeda.

🛡️ Aturan Emas API Key

  1. JANGAN taruh API Key di frontend (JavaScript, HTML). Browser = publik. Semua orang bisa lihat via F12.
  2. Simpan di server-side saja — appsettings.json, environment variable, atau secret manager.
  3. Jangan commit ke Git — tambahkan ke .gitignore. Bot bakal nemu dalam hitungan menit kalau bocor ke GitHub.
  4. Bisa di-rotate — kalau bocor, generate baru, yang lama di-revoke.
  5. Satu key, satu tujuan — beda service beda key, biar gampang dimatiin kalau ada masalah.

API Key vs Cara Autentikasi Lain

CaraCara KerjaKapan Dipakai
API KeyString statis, dikirim di headerSimple, server-to-server
JWT TokenToken sementara (punya expiry), hasil loginUser-based access

📌 API Key = paling sederhana. Cocok untuk mulai. Kalau nanti butuh lebih aman (mis. webhook nilai ujian yang tidak boleh bisa dipalsukan), baru upgrade ke API Key + HMAC Signature (kombinasi, bukan ganti).

Langkah 2 — Panggil API dari C# (kamu = konsumen)

public class WhatsAppService
{
    private readonly HttpClient _http;
    private readonly IConfiguration _config;

    public WhatsAppService(HttpClient http, IConfiguration config)
    {
        _http = http;
        _config = config;
    }

    public async Task KirimNotif(string nomorHp, string pesan)
    {
        var apiKey = _config["Fonnte:ApiKey"];
        var url = _config["Fonnte:BaseUrl"] + "/send";

        var request = new HttpRequestMessage(HttpMethod.Post, url);
        request.Headers.Add("Authorization", apiKey);  // ← di sinilah key dipakai

        request.Content = new FormUrlEncodedContent(new[]
        {
            new KeyValuePair<string, string>("target", nomorHp),
            new KeyValuePair<string, string>("message", pesan)
        });

        var response = await _http.SendAsync(request);
        response.EnsureSuccessStatusCode();
    }
}

Generate & Rotate API Key

Cara bikin, simpan aman, dan ganti API Key berkala tanpa downtime

1. Cara Generate API Key

Prinsip:

  • Harus random cryptographic — bukan Guid.NewGuid() (bisa ditebak polanya), bukan Random (pseudo-random, bisa diprediksi).
  • Cukup panjang — minimal 32 byte acak = 256 bit entropy.
  • URL-safe — pakai Base64Url atau hex.
  • Kasih prefix — biar gampang dikenali saat debug. Stripe pakai sk_live_, sk_test_.

Kode C# untuk Generate

using System.Security.Cryptography;

public static class ApiKeyGenerator
{
    public static string Generate(string prefix = "phc_live")
    {
        // 32 byte random = 256 bit, sangat cukup
        var bytes = RandomNumberGenerator.GetBytes(32);

        // Base64Url: tidak ada karakter '+', '/', '=' yang problematic di URL
        var token = Convert.ToBase64String(bytes)
            .Replace("+", "-")
            .Replace("/", "_")
            .TrimEnd('=');

        return $"{prefix}_{token}";
    }
}

Contoh hasil:

phc_live_J8xK2mP9qR4nL7vT3wY6jA1dF5hG8bN0cVzM4pQ

Strategi Prefix untuk PortalHC KPB

PrefixUntukKenapa?

phc_dev_Development lokalKalau ke-log pun tidak bahaya
💡 Kalau dev sekali tempel production key ke console log, langsung keliatan dari prefix-nya ⚠️.

2. Cara Simpan API Key di Database

⚠️ JANGAN simpan plaintext. Kalau DB kamu bocor atau backup jatuh ke tangan salah, semua key langsung bisa dipakai. Treat API Key seperti password.

Dua Pilihan Penyimpanan

Opsi A — Hash (satu arah) ⭐

  • Simpan Hash(key) di DB
  • Saat verifikasi: hash incoming → bandingkan
  • Key TIDAK bisa dilihat lagi setelah generate
  • Partner harus simpan sendiri
  • Rekomendasi untuk PortalHC

Opsi B — Enkripsi (dua arah)

  • Simpan Encrypt(key) pakai master key
  • Admin bisa lihat key lagi kalau lupa
  • Kalau master key bocor → semua key bobol
  • Lebih ribet ops

Kode Hash + Verify (Opsi A — HMAC-SHA256 cepat)

BCrypt itu aman tapi lambat (~50-100ms). Untuk webhook yang dipanggil setiap ujian selesai, pakai HMAC-SHA256 dengan pepper global lebih cocok:

using System.Security.Cryptography;
using System.Text;

public class ApiKeyService
{
    private readonly string _pepper; // rahasia global, di appsettings

    // Hash cepat tapi masih aman (karena plaintext key sudah 256 bit random)
    public string Hash(string key)
    {
        using var hmac = new HMACSHA256(Encoding.UTF8.GetBytes(_pepper));
        var hash = hmac.ComputeHash(Encoding.UTF8.GetBytes(key));
        return Convert.ToHexString(hash);
    }
}

Skema Tabel ApiKeys untuk PortalHC

CREATE TABLE ApiKeys (
    Id            INT PRIMARY KEY IDENTITY,
    ProviderId    INT NOT NULL,
    KeyHash       VARCHAR(64) NOT NULL,       -- HMAC-SHA256 hex
    KeyPreview    VARCHAR(8)  NOT NULL,       -- 4 char terakhir, untuk UI

    RevokedAt     DATETIME NULL,              -- soft delete

4. Kenapa & Kapan Harus Rotate?

Rotasi Terjadwal

Makin lama key hidup, makin besar peluang bocor. Best practice:

  • 90 hari — standar bank/fintech
  • 1 tahun — standar integrasi internal
  • Minimum sekali setahun — kalau tidak sempat sering, ini batas terakhir

Rotasi Event-Driven (WAJIB segera)

  • 🚨 Laptop yang nyimpan key hilang/dicuri
  • 🚨 Karyawan partner resign yang punya akses key
  • 🚨 Ketahuan bocor (commit ke GitHub publik, dsb)
  • 🚨 Audit finding mengindikasikan penyalahgunaan

5. Pattern "Grace Period" — Zero-Downtime Rotation

⚠️ Kalau kamu revoke key lama langsung begitu generate yang baru, partner putus koneksi sampai dia update config. Solusinya: dua key valid bersamaan selama masa transisi.

2

3

4

Key A RevokedAt = now. Request pakai Key A mulai ditolak 401.

Kode Verifikasi Multi-Key

public async Task<Provider?> ValidateApiKey(string incomingKey)
{
    var hash = Hash(incomingKey);

    var apiKey = await _db.ApiKeys
        .Include(k => k.Provider)
        .Where(k => k.KeyHash == hash)
        .Where(k => k.RevokedAt == null)                             // belum dicabut
        .Where(k => k.ExpiresAt == null || k.ExpiresAt > DateTime.UtcNow) // belum expired
        .FirstOrDefaultAsync();

    if (apiKey == null) return null;

    // Update last used (biar admin bisa track key mana yang masih aktif)
    apiKey.LastUsedAt = DateTime.UtcNow;
    await _db.SaveChangesAsync();

    return apiKey.Provider;
}

6. Checklist Wajib Sebelum Go-Live

  • ✅ Audit log — setiap create/revoke/rotate key ter-log (siapa, kapan, IP mana)
  • ✅ Alert email — kalau key stop dipakai >24 jam (partner lupa update) atau ada request pakai key revoked (indikasi serangan)
  • ✅ UI "last used" — admin bisa lihat semua key, kapan terakhir dipakai, dari IP mana
  • ✅ Incident runbook — dokumen 1 halaman untuk "key bocor → langkahnya"
  • ✅ Backup channel ke partner — nomor HP darurat kalau harus emergency rotate

Integrasi Assessment Eksternal

Studi kasus: peserta ujian di-redirect ke website partner, nilai balik via webhook

Skenario

💭 Cerita: "PortalHC saya terkait assessment. Kedepannya ada jenis assessment yang perlu website yang dibuat teman saya. Saya butuh peserta ketika ujian menekan tombol → redirect ke website teman. Setelah ujian, data dan nilai dikirim ke website saya."

Ini kasus klasik integrasi antar sistem via API + redirect flow — mirip SSO Login, Payment Gateway (Midtrans/Xendit), atau embed Zoom di LMS.

Keputusan Desain

Siapa Pegang Data Master? → PortalHC Induk, Partner = Engine Ujian

Peserta sudah ada di PortalHC + LDAP Pertamina. Assessment sudah terhubung ke workflow CDP, coaching, sertifikat. Jadi PortalHC tetap jadi source of truth; partner hanya mesin eksekusi ujian.

✅ Pola ini sama dengan Payment Gateway (toko induk, Midtrans cuma mesin bayar) dan SSO Login (aplikasi induk, Google cuma verifikasi).

Identifikasi Peserta di Sisi Partner? → Login Akun Terpisah

Peserta punya akun sendiri di web partner (dibuat via mapping NIP saat pertama redirect). Lebih ribet sedikit untuk user, tapi tanggung jawab lebih jelas dan auditable.

Aliran End-to-End

Fase 1 — Mulai Ujian

[Peserta di PortalHC]
   │ klik "Mulai Ujian Eksternal" pada assessment X
   ▼
[PortalHC Backend]
   │ (a) validasi: peserta eligible ikut assessment X?
   │ (b) generate ExternalExamSession:
   │       - SessionId (GUID)
   │       - Nip, AssessmentId, ExpiresAt (15 menit)
   │       - Token (JWT signed pakai shared secret)
   │       - Status = "Pending"
   │ (c) simpan ke DB
   │ (d) redirect browser ke:
   │     https://web-teman.com/exam/start?token=<JWT>
   ▼
[Web Partner — /exam/start]
   │ (e) decode JWT, verifikasi signature & expiry
   │ (f) baca claim: nip, assessmentId, sessionId
   │ (g) cek apakah NIP sudah punya akun di partner
   │       - Belum? Arahkan ke halaman registrasi/mapping
   │       - Sudah? Login otomatis sebagai user itu
   │ (h) (opsional) call-back ke PortalHC untuk confirm sessionId valid
   │ (i) mulai ujian
   ▼
[Peserta ngerjain ujian di web partner]

Fase 2 — Selesai Ujian

[Web Partner] submit → hitung nilai
   │
   │ (j) POST hasil ke PortalHC (webhook, server-to-server):
   │     POST /api/external/assessment/result
   │     Header: X-API-Key: <key-milik-partner>
   │     Header: X-Signature-SHA256: <HMAC dari body>
   │     Body:   { sessionId, nip, assessmentId, score, detail[], finishedAt }
   ▼
[PortalHC Backend]
   │ (k) validasi API Key
   │ (l) verifikasi signature HMAC (anti-tamper)
   │ (m) cek sessionId match + status masih "Pending"
   │ (n) simpan nilai ke tabel AssessmentResult
   │ (o) update session → Status = "Completed"
   │ (p) balas 200 OK ke web partner
   │
   │ (q) redirect peserta browser balik:
   │     https://portal-hc/Assessment/Finish?sessionId=xxx
   ▼
[Peserta lihat hasil di PortalHC]
💡 Kenapa ada step (l) HMAC di atas API Key? API Key saja bisa dicuri dan dipakai kirim nilai palsu. HMAC menandatangani isi body pakai shared secret — kalau body diubah 1 huruf, signature invalid. Ini praktik standar webhook (GitHub, Stripe, Midtrans pakai ini).

💻 Contoh Kode Endpoint Penerima Nilai

[ApiController]
[Route("api/external/assessment")]
public class ExternalAssessmentController : ControllerBase
{
    private readonly IApiKeyValidator _validator;
    private readonly AppDbContext _db;

    [HttpPost("result")]
    public async Task<IActionResult> TerimaNilai(
        [FromHeader(Name = "X-API-Key")] string apiKey,
        [FromBody] HasilUjianDto dto)
    {
        // 1. Validasi API Key
        var client = await _validator.Validate(apiKey);
        if (client == null)
            return Unauthorized(new { error = "Invalid API Key" });

        // 2. Validasi data
        if (string.IsNullOrEmpty(dto.Nip) || dto.Score < 0 || dto.Score > 100)
            return BadRequest(new { error = "Data tidak valid" });

        // 3. Cek session match + belum completed
        var session = await _db.ExternalExamSessions
            .FindAsync(dto.SessionId);
        if (session == null) return NotFound();
        if (session.Status == "Completed") return Conflict();

        // 4. Simpan hasil + update session
        _db.AssessmentResults.Add(new AssessmentResult
        {
            Nip = dto.Nip,
            AssessmentId = dto.AssessmentId,
            Score = dto.Score,
            SubmittedAt = DateTime.UtcNow,
            Source = client.ClientName,
            SourceSessionId = dto.SessionId
        });
        session.Status = "Completed";
        session.Score = dto.Score;
        session.CompletedAt = DateTime.UtcNow;

        await _db.SaveChangesAsync();

        return Ok(new { status = "received", id = dto.AssessmentId });
    }
}

🧪 Skenario Testing yang Wajib

  1. Happy path — redirect, ujian selesai, nilai masuk ✅
  2. Token expired — peserta idle 20 menit, partner harus reject
  3. Double submit — webhook dikirim 2x, hanya 1 yang masuk
  4. Tampered payload — ubah score setelah signature dihitung, ditolak 403
  5. Wrong API Key — ditolak 401
  6. Peserta tutup browser di tengah — session tetap Pending → resume atau expired otomatis
  7. Web partner down — PortalHC tampil error jelas, tidak freeze

🎯 Ruang Lingkup (YAGNI)

✅ Masuk Scope Awal

  • Satu provider (bisa diperluas nanti)
  • JWT redirect + API Key + HMAC webhook
  • Session TTL + idempotency
  • UI admin kelola API Key

⏭️ Nanti (Kalau Butuh)

  • Dashboard multi-provider
  • Real-time progress peserta
  • Live proctoring / anti-cheat video
  • Upgrade ke OAuth 2.0

🎓 Poin pembelajaran kunci dari studi kasus ini: Redirect + Webhook + JWT Token + API Key + HMAC — kombinasi ini adalah pola standar industri yang sama dipakai oleh Midtrans, Stripe, GitHub, dan sistem SSO. Kalau kamu paham kasus ini, kamu bisa pakai pola yang sama untuk integrasi masa depan apapun (payment, SSO, e-signature, dsb).
