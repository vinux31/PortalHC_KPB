Hybrid Mode: Semua user di-autentikasi via AD terlebih dahulu. Jika AD gagal, hanya akun admin@pertamina.com yang mendapat fallback ke login lokal. User lain akan gagal login jika AD tidak merespons.

3. Cara Mengaktifkan Active Directory

Opsi A: Via appsettings.json

1 Buka file appsettings.json di root project
2 Ubah nilai UseActiveDirectory dari false menjadi true: 
{
  "Authentication": {
    "UseActiveDirectory": true,
    "LdapPath": "LDAP://OU=KPB,OU=KPI,DC=pertamina,DC=com",
    "LdapTimeout": 5000,
    "AttributeMapping": {
      "Email": "mail",
      "FullName": "displayName",
      "NIP": "employeeID"
    }
  }
}
3 Restart aplikasi (bukan hot-reload — harus restart penuh karena service registration terjadi saat startup)

Opsi B: Via Environment Variable

Tanpa mengubah file, set environment variable:

# Windows (Command Prompt)
set Authentication__UseActiveDirectory=true

# Windows (PowerShell)
$env:Authentication__UseActiveDirectory = "true"

# Linux / Docker
export Authentication__UseActiveDirectory=true
Prioritas: Environment variable akan meng-override nilai di appsettings.json. Cocok untuk deployment tanpa mengubah file konfigurasi.

5. Hal-Hal yang Perlu Diperhatikan

5.1 Prasyarat Server

  • Server harus tergabung dalam domain (domain-joined) atau memiliki konektivitas jaringan ke Domain Controller.
  • Port LDAP (389) atau LDAPS (636) harus terbuka dari server aplikasi ke Domain Controller.
  • Jika menggunakan Docker, container harus memiliki akses jaringan ke AD server.

5.2 Data User Harus Sinkron

Penting: User harus terdaftar di kedua sistem — baik di Active Directory maupun di database Portal HC. AD hanya memverifikasi password; role, unit, dan data profil tetap dari database lokal.
  • Email di AD (mail attribute) harus sama persis dengan email di tabel AspNetUsers.
  • Jika user ada di AD tapi tidak ada di database Portal, login akan gagal.
  • Jika user ada di database tapi tidak ada di AD, login juga akan gagal (kecuali admin@pertamina.com).

5.3 Akun Admin Fallback

  • Hanya admin@pertamina.com yang memiliki fallback ke login lokal saat AD aktif.
  • Ini adalah safety net agar admin tetap bisa masuk meskipun AD server down.
  • Akun admin lain tidak mendapat fallback — hanya email tersebut yang di-hardcode di HybridAuthService.

5.4 Keamanan LDAP

  • LdapAuthService sudah memiliki proteksi LDAP injection via method EscapeLdapFilterValue.
  • Autentikasi dilakukan dengan user credential bind (bukan service account) — password user dikirim langsung ke AD untuk validasi.
  • Disarankan menggunakan LDAPS (LDAP over SSL, port 636) di production untuk mengenkripsi transmisi password.

5.5 Timeout & Error Handling

  • Timeout 5 detik mencegah login hang jika AD server tidak merespons.
  • Pesan error ditampilkan dalam Bahasa Indonesia (“Gagal terhubung ke server autentikasi”, dll.).
  • Error AD di-log ke ILogger — periksa log aplikasi jika ada masalah login.

5.6 LdapPath Harus Tepat

Perhatian: Jika LdapPath tidak sesuai dengan struktur Organizational Unit di AD, tidak ada user yang akan bisa login. Pastikan path sudah diverifikasi oleh tim IT/Infrastructure sebelum diaktifkan.

7. Rollback (Kembali ke Login Lokal)

Jika terjadi masalah setelah mengaktifkan AD:

1 Ubah UseActiveDirectory kembali ke false di appsettings.json
2 Atau hapus environment variable Authentication__UseActiveDirectory
3 Restart aplikasi — semua user kembali login dengan password lokal (Identity)
Tidak ada data yang hilang. Mengaktifkan/menonaktifkan AD tidak mengubah data di database. Hanya mekanisme verifikasi password yang berubah.
