1. Context — Kenapa Cleanup Diperlukan

Bug: Worker submit 1 assessment biasa (non-essay, non-PreTest) di Portal HC KPB → halaman /CMP/Records tampil 2 row untuk event sama: 1 row tipe "Assessment Online" + 1 row tipe "Training Manual" dengan judul prefix "Assessment: ". User report sebagai duplicate.

Root cause: Code di GradingService.GradeAndCompleteAsync() auto-generate baris TrainingRecord copy setiap kali worker selesai assessment. Halaman /CMP/Records via WorkerDataService.GetUnifiedRecords() union baris dari AssessmentSession + TrainingRecord → user lihat 2 baris untuk 1 event.

Regression history:

79284609 2026-03-18 REMOVE auto-create karena commit msg: "caused duplicate entries in RecordWorkerDetail unified view"
766011b6 2026-04-10 RE-ADD auto-create dengan dead-code try/catch DbUpdateException guard — regression yang Phase 324 fix
3023c5e7 2026-05-26 Plan 01 fix: hapus block insert TR di 3 lokasi production (GradingService + AssessmentAdminController)

Outcome: Setelah deploy code commit 3023c5e7 + apply SQL cleanup script ini → halaman /CMP/Records tampil 1 row per assessment event (sole source-of-truth = AssessmentSession).

Validation Lokal Done

Developer sudah validate cleanup di DB lokal: 18 row legacy TR hapus, AssessmentSessions Completed=28 utuh. Pre-fix + post-fix screenshot tersimpan di docs/screenshots/phase324/ sebagai proof.

2. Pre-Check Sample (MANDATORY sebelum DELETE)

Wajib — Sanity Inspection

IT JANGAN execute DELETE sebelum confirm sample 20 row semua auto-generated (Penyelenggara="Internal", Kategori match Assessment pattern, Judul prefix "Assessment: "). Kalau ada anomali (Penyelenggara berbeda, Kategori asing, Judul typo) → STOP dan eskalasi ke developer (Rino).

Query sample 20 row (sesuai filter yang dipakai cleanup script):

SELECT TOP 20
    Id, UserId,
    LEFT(Judul, 50) AS Judul,
    Kategori, Tanggal, TanggalSelesai,
    Penyelenggara, Status
FROM TrainingRecords
WHERE Judul LIKE 'Assessment:%'
  AND TanggalSelesai >= '2026-04-10'
ORDER BY Id DESC;

Expected sample pattern (validasi lokal Phase 324 Plan 03 Task 2):

  • Penyelenggara = 'Internal' (auto-create selalu Internal)
  • Kategori ∈ {'Assessment', 'OJT', 'Assessment OJT', dst — match assessment category nama yang ada di tabel AssessmentCategories}
  • Judul prefix "Assessment: " diikuti judul assessment original
  • TanggalSelesai populated (set di code line 273 GradingService pre-fix = DateTime.UtcNow saat insert)

Catatan: validasi lokal menemukan 0 row di lokal yang match filter TanggalSelesai >= '2026-04-10' (data lokal era pre-Mar-18 ASSESS-08). Di Dev/Prod kemungkinan ada row yang ter-insert antara 10 Apr - 26 Mei 2026 (selama bug regression aktif). Kalau IT temukan 0 row di Dev/Prod via query di atas, cleanup = no-op (script aman dijalankan, COMMIT silent).

2b. Orphan Pre-Check (defensive)

Phase 200 menambahkan RenewsTrainingId FK (OnDelete(NoAction)). Kalau ada TR child yang reference parent TR auto-generated, DELETE parent akan gagal dengan FK violation. Pre-check orphan count:

SELECT COUNT(*) AS OrphanRiskCount
FROM TrainingRecords
WHERE RenewsTrainingId IN (
    SELECT Id FROM TrainingRecords
    WHERE Judul LIKE 'Assessment:%'
      AND TanggalSelesai >= '2026-04-10'
);
  • Result = 0: Lanjut langsung Step 4 (script standard, null-clear pre-step SKIPPED). Validasi lokal verifikasi orphan count = 0.
  • Result > 0: IT uncomment block null-clear di script (Section 3, line "3. SKIPPED" comment) sebelum execute. Tanpa null-clear, DELETE akan gagal FK violation.

3. SQL Cleanup Script (Embedded)

Verbatim copy dari docs/sql/cleanup-2026-05-26-trainingrecord-duplicates.sql (commit 43f00210):

-- Phase 324 D-04..D-06 cleanup: hapus TrainingRecord auto-generated antara 2026-04-10 sampai hari ini.
-- Sumber bug: commit 766011b6 (2026-04-10) re-introduce auto-create TR yang menyebabkan visual duplicate di /CMP/Records.
-- Fix di code: Phase 324 Plan 01 hapus block insert TR di 3 lokasi production (GradingService + AssessmentAdminController).
-- Script ini idempotent: re-run aman, post-count = 0 = nothing to do.
-- Filter column: TanggalSelesai (TrainingRecords model TIDAK ada CreatedAt column - per RESEARCH A3 resolved).
-- Safety cap: 5000 row.
-- Database name: HcPortalDB_Dev (lokal). IT adjust ke HcPortalDB saat run di Prod via USE statement.
--
-- ENV NOTE:
-- - Script ini PRIMARY untuk Dev/Prod (data post-regression Apr 10).
-- - Untuk LOKAL recovery legacy data pre-Mar-18 (era ASSESS-08), pakai inline command dengan filter
--   pattern-only (no date filter). Lihat docs/SEED_JOURNAL.md entry Phase 324 untuk detail.

USE HcPortalDB_Dev;
GO

SET XACT_ABORT ON;
SET NOCOUNT ON;

BEGIN TRY
    BEGIN TRANSACTION cleanup324;

    -- 1. Pre-count
    DECLARE @preCount INT;
    SELECT @preCount = COUNT(*)
    FROM TrainingRecords
    WHERE Judul LIKE 'Assessment:%' AND TanggalSelesai >= '2026-04-10';
    PRINT CONCAT('Phase 324 cleanup pre-count: ', @preCount);

    -- 2. Safety cap (defensive guard)
    IF @preCount > 5000
    BEGIN
        ROLLBACK TRANSACTION cleanup324;
        RAISERROR('Safety cap exceeded (%d > 5000). Investigate before bumping cap.', 16, 1, @preCount);
        RETURN;
    END

    -- 3. (SKIPPED - RESEARCH OQ #3 resolved: orphan count = 0 di lokal verified Task 2)
    --    Tidak perlu null-clear RenewsTrainingId pre-step karena tidak ada child TR yang
    --    reference parent TR auto-generated. Kalau IT temui orphan > 0 di Dev/Prod via
    --    pre-check query, uncomment block berikut:
    -- UPDATE TrainingRecords
    -- SET RenewsTrainingId = NULL
    -- WHERE RenewsTrainingId IN (
    --     SELECT Id FROM TrainingRecords
    --     WHERE Judul LIKE 'Assessment:%' AND TanggalSelesai >= '2026-04-10'
    -- );
    -- DECLARE @nullClearedCount INT = @@ROWCOUNT;
    -- PRINT CONCAT('Phase 324 null-cleared RenewsTrainingId in child rows: ', @nullClearedCount);

    -- 4. DELETE
    DELETE FROM TrainingRecords
    WHERE Judul LIKE 'Assessment:%' AND TanggalSelesai >= '2026-04-10';
    DECLARE @deletedCount INT = @@ROWCOUNT;
    PRINT CONCAT('Phase 324 cleanup deleted: ', @deletedCount);

    -- 5. Post-count verify
    DECLARE @postCount INT;
    SELECT @postCount = COUNT(*)
    FROM TrainingRecords
    WHERE Judul LIKE 'Assessment:%' AND TanggalSelesai >= '2026-04-10';
    PRINT CONCAT('Phase 324 cleanup post-count: ', @postCount);

    IF @postCount > 0
    BEGIN
        ROLLBACK TRANSACTION cleanup324;
        RAISERROR('Phase 324 post-count not zero (%d). Rolled back.', 16, 1, @postCount);
        RETURN;
    END

    COMMIT TRANSACTION cleanup324;
    PRINT 'Phase 324 cleanup committed successfully.';
END TRY
BEGIN CATCH
    IF XACT_STATE() <> 0 ROLLBACK TRANSACTION cleanup324;
    DECLARE @errMsg NVARCHAR(4000) = ERROR_MESSAGE();
    PRINT CONCAT('Phase 324 cleanup failed: ', @errMsg);
    THROW;
END CATCH;
GO

4. Checklist IT (URUTAN WAJIB)

URUTAN WAJIB — Cegah Race Condition

Step 1 deploy code DULU (commit 3023c5e7) supaya app berhenti generate TR baru. STEP 2 baru SQL cleanup. Jangan reverse order — race condition bisa bikin row baru ter-insert selama cleanup running.

  1. Step 1 — Deploy code commit 3023c5e7: 
    cd /path/to/PortalHC_KPB
git fetch origin main
git checkout 3023c5e7   # atau git pull origin main kalau sudah ada commit terbaru
# Restart IIS app pool (atau systemctl restart kalau Kestrel-native)
iisreset    # Windows; atau equivalent restart untuk env IT

    Verifikasi deploy sukses: cek log app start, akses http://10.55.3.3/KPB-PortalHC, login → halaman tidak error.

  2. Step 2 — BACKUP DB Dev: 
    BACKUP DATABASE HcPortalDB_Dev
TO DISK='C:\Backup\HcPortalDB_Dev.20260526-phase324-pre-cleanup.bak'
WITH FORMAT, INIT;

    Wajib backup sebelum DELETE — supaya rollback option tersedia (Section 7).

  3. Step 3 — Pre-check sample query (lihat Section 2):

    Execute query sample 20 row + orphan check. Visual inspect output. Kalau ada anomali → STOP dan eskalasi ke developer.

  4. Step 4 — Execute cleanup script: 
    sqlcmd -S "10.55.3.3" -U sa -P "<password>" -C \
       -i "C:\path\to\cleanup-2026-05-26-trainingrecord-duplicates.sql"

    (Adjust connection string sesuai env IT. Untuk Windows Integrated Security pakai -E tanpa -U/-P.)

    Expected output:

    Phase 324 cleanup pre-count: <N>

Phase 324 cleanup post-count: 0
Phase 324 cleanup committed successfully.
  5. Step 5 — Post-verify count (lihat Section 5):

    Confirm post-cleanup count = 0 + AssessmentSessions count UTUH.

  6. Step 6 — Idempotency check (re-run script):

    Run script kedua kali. Expected: pre-count: 0, deleted: 0, post-count: 0, committed successfully. Kalau output beda → eskalasi.

RESTORE DATABASE HcPortalDB_Dev
FROM DISK='C:\Backup\HcPortalDB_Dev.20260526-phase324-pre-cleanup.bak'
WITH REPLACE;
ALTER DATABASE HcPortalDB_Dev SET MULTI_USER;
Prerequisite Rollback

Rollback hanya feasible kalau Step 2 BACKUP dilakukan. Tanpa .bak file, rollback impossible. Itulah kenapa Step 2 wajib + tidak boleh skip.

Kalau rollback dilakukan, eskalasi ke developer untuk investigate kenapa cleanup mis-target.
