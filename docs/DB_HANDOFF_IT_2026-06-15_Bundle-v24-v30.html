4. Self-heal Data Dev + SQL Cadangan (F1 & F2)

Otomatis — Idempotent

Saat app start, SeedData.InitializeAsync (Program.cs:146) menjalankan 2 self-heal idempotent (aman, no-op kalau data sudah benar). Normalnya IT cukup deploy — keduanya beres sendiri. SQL manual di bawah hanya cadangan kalau verifikasi Step 5 menunjukkan self-heal GAGAL (karena startup menelan error).

F1 — Normalize Level Organisasi

NormalizeOrganizationLevelsAsync (Data/SeedData.cs:197) menghitung ulang OrganizationUnits.Level berdasar kedalaman pohon (root = 0, anak = induk + 1), lalu hanya UPDATE baris yang levelnya beda. Idempotent, tidak menambah/hapus baris — hanya geser nilai kolom Level. Memperbaiki "split-brain" Level di Dev (data lama pakai 1/2, kode baru pakai 0/1).

SQL cadangan F1 (jalankan HANYA bila Step 5b tidak mulai dari 0)

Pakai recompute-by-depth — JANGAN blanket Level = Level - 1

SQL ini meniru persis logika kode (NormalizeOrganizationLevelsAsync, BFS tree-depth). Jangan pakai geser Level - 1 blanket: kalau Dev "split-brain" (sebagian unit sudah Level 0, sebagian masih 1/2), geser -1 justru merusak yang sudah benar. Recompute-by-depth menghitung kedalaman tiap node dari root, jadi aman untuk kondisi campur.

BEGIN TRANSACTION;

-- Recompute Level = kedalaman pohon (root ParentId IS NULL = 0, anak = induk + 1)
;WITH Tree AS (
    SELECT Id, 0 AS Depth
    FROM OrganizationUnits
    WHERE ParentId IS NULL
    UNION ALL
    SELECT ou.Id, t.Depth + 1
    FROM OrganizationUnits ou
    INNER JOIN Tree t ON ou.ParentId = t.Id
)
UPDATE ou
SET ou.Level = t.Depth
FROM OrganizationUnits ou
INNER JOIN Tree t ON ou.Id = t.Id
WHERE ou.Level <> t.Depth;     -- hanya baris yang beda (idempotent)

-- verifikasi: Level terkecil harus 0
SELECT Level, COUNT(*) AS Jml FROM OrganizationUnits GROUP BY Level ORDER BY Level;

-- Kalau benar (mulai 0) → COMMIT; kalau ragu → ROLLBACK;
COMMIT;

Idempotent (hanya UPDATE baris yang Level-nya beda). Unit orphan (tak terjangkau dari root) tidak tersentuh — sama dgn perilaku kode. Bila tabel sangat dalam dan kena batas rekursi, tambah OPTION (MAXRECURSION 1000) di akhir UPDATE; bila terindikasi cycle (error rekursi), eskalasi developer.

F2 — Isi Ulang ProtonTracks

SeedProtonTracksAsync (Data/SeedData.cs:143) memastikan 6 baris master Track ada (Panelman/Operator × Tahun 1/2/3). Logikanya insert-missing per-key (bukan all-or-nothing): tabel kosong → isi 6; tabel partial → top-up yang hilang. Memperbaiki dropdown Track kosong + StatusData [] di Dev.

Catatan — Self-heal TIDAK repair FK orphan

F2 hanya INSERT baris Track yang hilang; ia tidak menyambung ulang data anak lama (ProtonKompetensiList/ProtonTrackAssignments) yang menunjuk ProtonTrackId yang sudah terhapus — baris Track baru dapat Id baru. Dalam kasus Dev (tabel tadinya kosong) normalnya belum ada anak, jadi aman. Kalau ragu / ada gejala, eskalasi developer.

SQL cadangan F2 (jalankan HANYA bila Step 5a ≠ 6)

-- Idempotent + additive: MERGE WHEN NOT MATCHED (baris existing di-skip, tidak dobel)
WITH ExpectedTracks AS (
    SELECT 'Panelman' AS TrackType, 'Tahun 1' AS TahunKe, 'Panelman - Tahun 1' AS DisplayName, 1 AS Urutan
    UNION ALL SELECT 'Panelman', 'Tahun 2', 'Panelman - Tahun 2', 2
    UNION ALL SELECT 'Panelman', 'Tahun 3', 'Panelman - Tahun 3', 3
    UNION ALL SELECT 'Operator', 'Tahun 1', 'Operator - Tahun 1', 4
    UNION ALL SELECT 'Operator', 'Tahun 2', 'Operator - Tahun 2', 5
    UNION ALL SELECT 'Operator', 'Tahun 3', 'Operator - Tahun 3', 6
)
MERGE INTO ProtonTracks pt
USING ExpectedTracks et ON pt.TrackType = et.TrackType AND pt.TahunKe = et.TahunKe
WHEN NOT MATCHED THEN
    INSERT (TrackType, TahunKe, DisplayName, Urutan)
    VALUES (et.TrackType, et.TahunKe, et.DisplayName, et.Urutan);

-- verifikasi
SELECT Id, TrackType, TahunKe, DisplayName, Urutan FROM ProtonTracks ORDER BY Urutan;  -- Expected: 6 baris
Tidak Perlu Restart App Setelah SQL Cadangan

Kedua tabel dibaca per-request. Begitu data benar, halaman normal pada refresh berikutnya. Tidak perlu restart IIS / app pool.

5. Post-Deploy: Recompute Skor Essay

Prod/Dev yang masih jalan code lama bisa punya sesi essay-only yang sudah di-finalize tapi Score=0 (bug grading lama, sudah ke-fix di forward path bundle ini). Endpoint recompute memperbaiki baris historis itu.

EndpointPOST /Admin/RecomputeEssayScores
Authz[Authorize(Roles="Admin")] + [ValidateAntiForgeryToken] (Admin-only)
IdempotentYa — hanya sentuh kandidat Status='Completed' AND HasManualGrading=1 AND (Score IS NULL OR Score=0) yang semua essay-nya sudah dinilai. Run kedua = no-op.
DampakSet Score + IsPassed SAJA (D-03). TIDAK menerbitkan sertifikat / penanda Proton / notifikasi / TrainingRecord retroaktif. Status tidak diubah.

Langkah

  1. Count kandidat dulu: 
    SELECT COUNT(*) AS Kandidat FROM AssessmentSessions
WHERE Status='Completed' AND HasManualGrading=1 AND (Score IS NULL OR Score=0);
  2. Login Admin → trigger POST /Admin/RecomputeEssayScores (tombol UI bila ada, atau form POST / headless POST).
  3. Cek TempData response: {repaired} diperbaiki, {skipped} dilewati, {alreadyOk} sudah benar + audit log entry RecomputeEssayScores.
  4. Re-run query count → harus berkurang (idempotent, aman re-run).
Jangan UPDATE Manual

JANGAN repair Score dengan UPDATE manual — gunakan endpoint (auditable + logika agregasi identik forward path, anti-drift). Sertifikat untuk baris repaired yang lulus TIDAK auto-terbit; bila perlu, HC re-trigger per-orang manual.
