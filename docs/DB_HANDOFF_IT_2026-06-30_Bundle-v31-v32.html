4. ⚠️ Khusus Migration #422 (AddPackageNumberUniqueIndex)

Migration ini mengubah data existing, bukan cuma skema:

  • STEP 1 — auto-renumber PackageNumber per session via ROW_NUMBER() (gap-free 1..N), untuk membersihkan duplikat lama (bug penomoran count-based + delete tanpa renumber). Idempotent (re-run = hasil sama).
  • STEP 2 — bikin UNIQUE index IX_AssessmentPackages_SessionId_PackageNumber_Unique.
ImplikasiDetail
✅ Tidak perlu fix duplikat manualSTEP 1 sudah menanganinya otomatis. Migration tidak akan gagal karena ada duplikat lama.
⚠️ Renumber permanenDown() tidak me-revert renumber. Nomor paket yang tampil pada session lama bisa berubah (gap-free).
⚠️ Backup WAJIBKarena ada mutasi data permanen (selain SOP standar). Satu-satunya jalan balik penuh = restore .bak.

5. Cara Apply + Checklist IT (URUTAN WAJIB)

Penting — migration auto-apply, tapi error-nya ditelan

Program.cs:151 memanggil context.Database.Migrate() saat startup → 6 migration apply otomatis begitu build baru jalan. TAPI blok itu dibungkus try/catch yang hanya nge-log error (app tetap start walau migrasi gagal). Maka WAJIB verifikasi migration benar-benar ter-apply (Step 4–5) — jangan cukup percaya "app-nya jalan".

Urutan langkah

  1. BACKUP DATABASE penuh (SQL di Section 4). Capture path .bak untuk rollback. WAJIB
  2. Pull / checkout main (2dff7435) ke server. git pull origin main.
  3. Build / publish ulang artifact + restart app (IIS / Kestrel pool). Saat start → 6 migration auto-apply.
  4. Verifikasi 6 migration ter-apply (query __EFMigrationsHistory): 
    SELECT MigrationId FROM __EFMigrationsHistory
WHERE MigrationId IN (
  '20260618045427_AddUserUnitsTable',
  '20260621011101_AddParticipantRemovalColumns',
  '20260621065918_AddRetakeColumnsAndArchive',
  '20260622124217_AddAssessmentPackageSection',
  '20260623103224_AddPackageNumberUniqueIndex',
  '20260624133656_AddTokenVerifiedAt'
)
ORDER BY MigrationId;
-- Expected: 6 baris. Kalau < 6 → migrasi gagal (cek log startup), JANGAN lanjut.
  5. Verifikasi skema baru ada: 
    -- Tabel BARU di bundle ini (expect 3 baris)
SELECT TABLE_NAME FROM INFORMATION_SCHEMA.TABLES
WHERE TABLE_NAME IN ('UserUnits','AssessmentPackageSections','AssessmentAttemptResponseArchives');

-- Kolom baru kunci (expect 4 baris)
SELECT TABLE_NAME, COLUMN_NAME FROM INFORMATION_SCHEMA.COLUMNS
WHERE (TABLE_NAME='PackageQuestions'   AND COLUMN_NAME='SectionId')
   OR (TABLE_NAME='AssessmentSessions' AND COLUMN_NAME IN ('TokenVerifiedAt','MaxAttempts','AllowRetake'));

-- Unique index #422 (expect 1 baris)
SELECT name FROM sys.indexes WHERE name = 'IX_AssessmentPackages_SessionId_PackageNumber_Unique';
  6. Smoke test (Section 6) → isi Log Pengerjaan IT (Section 10) → kabari developer.
  7. Dev OK → ulang ke Prod (backup → deploy → verifikasi sama).

9. Rollback Plan

Opsi A — Restore BACKUP (utama, karena #422 mutasi data)

USE master;
ALTER DATABASE HcPortalDB_Dev SET SINGLE_USER WITH ROLLBACK IMMEDIATE;
RESTORE DATABASE HcPortalDB_Dev FROM DISK = N'<backup-path>.bak' WITH REPLACE;
ALTER DATABASE HcPortalDB_Dev SET MULTI_USER;

Lalu redeploy build lama (baseline e5f342e5).

Opsi B — EF Down (skema saja; TIDAK me-revert renumber #422)

dotnet ef database update 20260613095102_AddShuffleTogglesToAssessmentSession --project HcPortal.csproj

Mengembalikan skema ke titik baseline 06-15. Catatan: data PackageNumber yang sudah di-renumber #422 TIDAK kembali ke nomor lama — untuk pemulihan data penuh pakai Opsi A.

⚠️ Opsi B sendirian PERCUMA kalau build baru masih ter-deploy. Program.cs:151 Database.Migrate() jalan tiap startup → restart app berikutnya akan auto-apply ulang ke-6 migration secara diam-diam. Supaya EF Down efektif: hentikan / re-deploy build lama (baseline e5f342e5) dulu, baru jalankan Down. Untuk rollback sebenarnya saat insiden, pakai Opsi A (restore .bak + redeploy build lama).
