User Impersonation / View As Role

Kompleksitas Tinggi
Prioritas Sedang
Apa Ini?

Kemampuan admin untuk "melihat" atau "masuk ke" akun user lain tanpa mengetahui password mereka. Admin bisa melihat persis apa yang dilihat oleh user tersebut — halaman, data, menu, tombol — seolah-olah sedang login sebagai user itu. Setelah selesai, admin kembali ke akun aslinya dengan satu klik.

Dua Pendekatan
AspekView As Role (Ringan)Full Impersonation (Lengkap)
Cara KerjaDropdown di navbar: "Lihat sebagai User" — preview tampilan role tertentuPilih user spesifik dari daftar, login sebagai user tersebut
ModeRead-only — tidak bisa melakukan aksiBisa melakukan aksi atas nama user (tergantung konfigurasi)
Data yang DilihatData generik/sample sesuai roleData asli milik user yang dipilih
KompleksitasRendah — hanya switch role di sessionTinggi — perlu audit trail, security layer, session management
Cocok UntukVerifikasi tampilan, testing UI per roleTroubleshooting masalah spesifik user, support ticket
Fungsi Utama
  • Troubleshoot masalah yang dilaporkan user tanpa perlu minta screenshot
  • Verifikasi bahwa permission dan tampilan benar untuk setiap role
  • Testing fitur baru dari perspektif user sebelum rilis
  • Support user yang kesulitan tanpa harus datang ke tempat mereka
  • Debug data-specific issues (contoh: "assessment saya tidak muncul")
Dampak & Manfaat
Efisiensi Support
Waktu troubleshoot berkurang 60-80%. Tidak perlu bolak-balik tanya screenshot ke user.
Quality Assurance
Admin bisa langsung verifikasi tampilan setiap role setelah update fitur.
Risiko Keamanan
Jika tidak diimplementasi dengan benar, bisa disalahgunakan. Wajib ada audit trail lengkap.
Effort Implementasi
View As Role: 2-3 hari. Full Impersonation: 1-2 minggu (termasuk security layer).
Contoh Skenario di PortalHC
Skenario 1: User lapor "saya tidak bisa lihat assessment"

Admin klik "Impersonate" di profil user → melihat halaman CMP persis seperti user → menemukan bahwa user belum di-assign ke assessment → kembali ke akun admin → assign user.

Skenario 2: Testing role HC setelah update fitur

Admin pilih "View as HC" → cek semua menu muncul dengan benar → verifikasi data yang tampil sesuai → switch kembali ke Admin.

Best Practice & Pengaman
  • Banner peringatan — Banner kuning/merah menyala selama impersonation aktif
  • Tombol "Kembali ke Admin" — Selalu visible, satu klik untuk kembali
  • Auto-expire — Session impersonation otomatis berakhir setelah 30 menit
  • Audit log — Setiap impersonation tercatat: siapa, kapan, user mana, durasi
  • Blokir aksi sensitif — Tidak bisa ubah password, role, atau setting saat impersonate
  • Tidak bisa impersonate role setara/lebih tinggi — Admin tidak bisa impersonate admin lain
Platform Referensi
HubSpot (30 min auto-expire) WordPress (View Admin As plugin) Google Ad Manager Salesforce Okta

Announcement / Broadcast

Kompleksitas Sedang
Prioritas Tinggi
Apa Ini?

Fitur bagi admin untuk mengirim pengumuman atau pesan broadcast ke seluruh user atau kelompok tertentu. Pengumuman muncul sebagai banner di atas halaman, pop-up saat login, atau item di halaman khusus pengumuman. Ini menjadi saluran komunikasi resmi dari admin ke semua pengguna sistem.

Fungsi Utama
  • Menginformasikan jadwal assessment mendatang ke seluruh peserta
  • Memberitahu perubahan kebijakan, prosedur, atau aturan baru
  • Mengingatkan deadline pengumpulan dokumen atau sertifikat
  • Mengumumkan maintenance terjadwal atau downtime
  • Broadcast informasi urgent (misalnya: perubahan jadwal ujian mendadak)
  • Targeting: kirim ke semua, per unit/bagian, per role, atau per individu
Jenis Tampilan Pengumuman
TipeTampilanCocok Untuk
BannerPita di atas halaman (bisa di-dismiss)Info penting tapi tidak mendesak (maintenance terjadwal)
Modal Pop-upDialog muncul saat login/buka appPengumuman wajib baca (perubahan kebijakan)
Halaman PengumumanDaftar pengumuman dengan read/unreadArsip lengkap, referensi kembali
Toast/AlertNotifikasi kecil di sudut layarInfo ringan, tidak mengganggu workflow
Dampak & Manfaat
Komunikasi Terpusat
Semua pengumuman di satu tempat. Tidak ada lagi "saya tidak tahu" karena info tercecer di WA group.
Tracking Terbaca
Admin bisa tahu siapa yang sudah dan belum membaca pengumuman. Bisa follow-up yang belum baca.
Mengurangi Beban HC
HC tidak perlu WA satu-satu ke pekerja. Cukup buat pengumuman, semua yang login pasti lihat.
Effort Implementasi
3-5 hari kerja: tabel announcements, CRUD admin, rendering di layout, read tracking.
Contoh Skenario di PortalHC
Skenario 1: Jadwal assessment

Admin buat pengumuman: "Assessment Operator akan dilaksanakan 15-17 April 2026. Pastikan semua peserta sudah mempelajari KKJ masing-masing." → Target: Unit Alkylation + RFCC NHT → Muncul sebagai banner saat login.

Skenario 2: Perubahan kebijakan

Admin buat pengumuman modal: "Mulai Mei 2026, semua sertifikat wajib di-upload dalam 7 hari setelah terbit." → Target: Semua user → Wajib klik "Saya mengerti" sebelum bisa lanjut.

Fitur yang Direkomendasikan
  • Rich text editor (bold, italic, link) untuk isi pengumuman
  • Lampiran file (PDF, gambar)
  • Penjadwalan publish (tanggal mulai & berakhir)
  • Target audience: semua, per bagian/unit, per role
  • Prioritas: Normal, Penting, Urgent (beda warna/ikon)
  • Status read/unread per user + laporan pembaca
  • Pin pengumuman (tetap di atas)
Platform Referensi
Asana (announcement-only channel) Microsoft Teams (org-wide notification) Moodle LMS (site announcements) Google Workspace (admin alerts)

In-App Notification

Kompleksitas Tinggi
Prioritas Tinggi
Apa Ini?

Sistem notifikasi di dalam aplikasi — ikon lonceng (🔔) di navbar yang menampilkan daftar pemberitahuan real-time. Setiap event penting di sistem otomatis menghasilkan notifikasi ke user terkait. User bisa melihat daftar notifikasi, menandai sudah dibaca, dan klik untuk langsung menuju halaman terkait.

Fungsi Utama
  • Notifikasi otomatis saat di-assign ke assessment baru
  • Reminder saat ujian hampir dimulai atau deadline mendekat
  • Pemberitahuan saat deliverable coaching di-approve atau di-reject
  • Alert saat sertifikat mendekati masa expired
  • Notifikasi saat ada pengumuman baru dari admin
  • Info saat coach memberikan feedback di sesi coaching
  • Badge counter (angka merah) menunjukkan jumlah unread
Perbedaan dengan Announcement
AspekAnnouncementIn-App Notification
Siapa yang buatAdmin/HC manualSistem otomatis
KontenPengumuman panjang, rich textPesan pendek, 1-2 kalimat
SifatSatu ke banyak (broadcast)Personal ke masing-masing user
Contoh"Assessment batch April dijadwalkan...""Anda telah di-assign ke Assessment Operator #12"
Dampak & Manfaat
User Engagement
User tidak perlu mengecek setiap halaman. Notifikasi membawa info ke mereka.
Tidak Ada yang Terlewat
Deadline, approval, assignment — semua event penting pasti ternotifikasi.
Mengurangi Ketergantungan WA
Saat ini banyak info disampaikan via WA/telepon. Notifikasi in-app mengurangi beban ini.
Risiko: Notification Fatigue
Terlalu banyak notifikasi = user abaikan semua. Harus dikurasi: hanya event penting.

  • Tabel Notifications (UserId, Title, Message, Link, IsRead, CreatedAt)
  • Icon lonceng di navbar dengan badge counter (unread count)
  • Dropdown panel saat klik lonceng — daftar notifikasi terbaru
  • Halaman "Semua Notifikasi" dengan filter dan pagination
  • "Tandai semua sebagai dibaca" button
  • Optional: polling setiap 60 detik untuk cek notifikasi baru (atau SignalR untuk real-time)

Soft Delete / Recycle Bin

Kompleksitas Sedang
Prioritas Sedang
Apa Ini?

Saat admin menghapus data (user, assessment, soal, dll), data tidak benar-benar dihapus dari database, melainkan ditandai sebagai "dihapus" (soft delete). Data yang dihapus masuk ke Recycle Bin — halaman khusus dimana admin bisa melihat semua item yang terhapus dan me-restore jika terhapus karena kesalahan. Setelah 30 hari, data otomatis dihapus permanen (hard delete).

Fungsi Utama
  • Mencegah kehilangan data akibat kesalahan hapus (human error)
  • Menyediakan "jaring pengaman" untuk operasi destruktif
  • Admin bisa restore data tanpa perlu kontak developer/DBA
  • Recycle Bin sebagai staging area sebelum penghapusan permanen
  • Auto-purge setelah retention period (misal 30 hari)
Cara Kerja
TahapApa yang Terjadi
1. Admin klik "Hapus"Data ditandai IsDeleted = true, DeletedAt = DateTime.Now. Data hilang dari tampilan normal tapi masih ada di database.
2. Masuk Recycle BinData muncul di halaman Recycle Bin. Admin bisa lihat apa saja yang terhapus.
3. Restore (opsional)Admin klik "Restore" → data kembali aktif, IsDeleted = false.
4. Auto-purgeSetelah 30 hari, background job menghapus permanen dari database.
Dampak & Manfaat
Keamanan Data
Tidak ada lagi insiden "data hilang karena tidak sengaja dihapus". Bisa selalu di-restore.
Kemandirian Admin
Admin bisa self-service restore tanpa menghubungi developer untuk restore dari backup.
Compliance
Mendukung data retention policy. Data terhapus tetap tersedia selama periode tertentu.
Ukuran Database
Data yang di-soft delete masih menempati storage. Perlu auto-purge untuk menjaga ukuran DB.
Entitas yang Perlu Soft Delete di PortalHC
EntitasAlasanDampak Jika Hilang
AssessmentBerisi soal, peserta, hasil — data kompleksTinggi — kehilangan riwayat ujian
Paket SoalSoal bisa ratusan, butuh waktu lama dibuatTinggi — harus import ulang
Data PekerjaSudah ada Deactivate, tapi delete masih hardSedang — kehilangan riwayat training
SertifikatDokumen resmi, tidak boleh hilang sembaranganTinggi — bukti kompetensi hilang
Rekam PelatihanRiwayat training penting untuk auditSedang — kehilangan catatan pelatihan
Implementasi Teknis
  • Tambah kolom IsDeleted, DeletedAt, DeletedBy di setiap entitas
  • Global query filter di EF Core: .HasQueryFilter(e => !e.IsDeleted)
  • Halaman Recycle Bin: daftar item terhapus, grouped by tipe entitas
  • Tombol Restore dan Hapus Permanen per item
  • Background job (Hangfire/hosted service) untuk auto-purge 30 hari
Platform Referensi
WordPress (Trash — 30 hari) Google Drive (Sampah — 30 hari) Salesforce (Recycle Bin — 15 hari) SharePoint (Site Recycle Bin)

Dashboard Statistik Admin

Kompleksitas Sedang
Prioritas Tinggi
Apa Ini?

Halaman dashboard khusus admin yang menampilkan ringkasan statistik dan KPI seluruh sistem dalam satu tampilan. Saat admin login, langsung terlihat overview: berapa pekerja aktif, assessment berjalan, sertifikat expiring, progress coaching, dan metrik penting lainnya — tanpa harus membuka satu per satu halaman.

KPI & Metrik yang Ditampilkan
KategoriMetrikVisualisasi
PekerjaTotal pekerja aktif vs non-aktifAngka besar + badge

Distribusi per roleDonut chart
AssessmentAssessment aktif (sedang berjalan)Angka + status badge
Tingkat kelulusan bulan iniPersentase + trend arrow
Assessment terjadwal mendatangMini calendar / list
SertifikatSertifikat yang akan expired (30/60/90 hari)Alert card berwarna
Total sertifikat aktif vs expiredPie chart
CoachingProgress coaching keseluruhanProgress bar
Bottleneck approval (pending > 7 hari)Alert list
SistemUser login hari iniAngka
Aktivitas terbaru (audit log singkat)Timeline
Dampak & Manfaat
Visibilitas Instan
Admin langsung tahu kondisi terkini tanpa harus membuka 10 halaman berbeda.
Deteksi Masalah Dini
Sertifikat expiring, approval bottleneck, assessment gagal — langsung terlihat sebelum jadi masalah besar.
Data-Driven Decision
Manajemen bisa ambil keputusan berdasarkan data real, bukan estimasi.
Effort Implementasi
3-5 hari: query aggregation, chart library (Chart.js), card layout, caching.
Contoh Skenario di PortalHC
Skenario: Admin buka dashboard pagi hari

Langsung terlihat: 3 sertifikat expired minggu ini (alert merah), 2 assessment berjalan, 15 approval coaching pending >7 hari (bottleneck). Admin langsung tahu harus follow-up apa hari ini.

Fitur yang Direkomendasikan
  • Filter periode: hari ini, minggu ini, bulan ini, custom range
  • Filter unit/bagian untuk fokus area tertentu
  • Quick action buttons: langsung jump ke halaman terkait dari dashboard
  • Auto-refresh setiap 5 menit
  • Export dashboard sebagai PDF (untuk laporan manajemen)
  • Widget yang bisa di-reorder (nice-to-have)
Platform Referensi
Google Admin Console HubSpot Dashboard Moodle Site Admin WordPress Dashboard Metabase

Kompleksitas Sedang
Prioritas Sedang
Apa Ini?

Halaman pengaturan terpusat dimana admin bisa mengubah konfigurasi sistem tanpa harus edit code atau config file. Semua parameter yang sering berubah — nama aplikasi, logo, batas waktu, aturan password, dll — bisa diubah dari UI oleh admin. Perubahan langsung berlaku tanpa restart atau deploy ulang.

Mengurangi Deploy

Contoh Skenario di PortalHC
Skenario: Ubah passing score assessment

Manajemen putuskan passing score naik dari 70% ke 75%. Admin buka Settings → Assessment → ubah Passing Score → Save. Semua assessment baru otomatis pakai 75%. Tidak perlu kontak developer.

Skenario: Ganti logo perusahaan

Perusahaan rebranding. Admin buka Settings → Umum → Upload logo baru → Save. Logo di seluruh halaman langsung berubah.

Implementasi Teknis
  • Tabel SystemSettings (Key, Value, Category, DataType, Description)
  • Service ISettingsService dengan caching (MemoryCache/Redis)
  • UI: halaman tab per kategori, form input sesuai tipe data
  • Validasi: min/max, required, tipe data, confirmation sebelum save
  • Audit log: catat setiap perubahan setting

Maintenance Mode

Kompleksitas Rendah
Prioritas Sedang
Apa Ini?

Kemampuan admin untuk menonaktifkan akses user ke website secara sementara selama proses pemeliharaan (update database, deploy versi baru, migrasi data, dll). User yang mencoba akses akan melihat halaman "Sedang Dalam Pemeliharaan" yang informatif, sementara admin tetap bisa login dan bekerja.

Fungsi Utama
  • Toggle on/off maintenance mode dari admin panel (satu klik)
  • Halaman maintenance kustom: pesan, estimasi waktu selesai, contact info
  • Admin & HC tetap bisa login selama maintenance (bypass)
  • Penjadwalan maintenance: set waktu mulai & selesai otomatis
  • Auto-disable maintenance mode saat waktu selesai tercapai
  • User yang sedang login saat maintenance diaktifkan: redirect ke halaman maintenance
Dampak & Manfaat
Profesional
Daripada user melihat error 500 saat update, mereka lihat halaman informatif yang rapi.
Keamanan Update
Mencegah user mengakses data saat database sedang dalam proses migrasi (risiko data corrupt).
Kontrol Admin
Admin bisa verifikasi update berhasil sebelum membuka akses ke semua user.
Effort Implementasi
1-2 hari saja: middleware check, halaman maintenance, toggle di settings.
Contoh Skenario di PortalHC
Skenario: Deploy versi baru

Admin aktifkan maintenance mode → Deploy versi baru → Test cepat oleh admin → Maintenance mode off → User bisa akses lagi. Total downtime hanya yang benar-benar diperlukan.

Skenario: Maintenance terjadwal

Admin jadwalkan maintenance Sabtu 22:00 - Minggu 02:00. Pengumuman otomatis muncul H-1. Saat waktunya tiba, sistem otomatis masuk maintenance mode. Pukul 02:00 otomatis kembali normal.

Tampilan Halaman Maintenance
  • Logo perusahaan
  • Pesan: "Sistem sedang dalam pemeliharaan"
  • Estimasi waktu selesai
  • Progress bar (opsional)
  • Contact person untuk keadaan darurat
  • Design yang profesional dan menenangkan
Implementasi Teknis
  • Middleware ASP.NET Core: cek flag maintenance, bypass untuk role Admin/HC
  • Setting di database: MaintenanceMode (bool), MaintenanceMessage, MaintenanceEndTime
  • Halaman static maintenance (tidak bergantung database — fallback)
  • Toggle di halaman System Settings
Platform Referensi
WordPress (maintenance-mode plugin) Laravel (php artisan down) Shopify (password-protected storefront) GitHub (status page)

Backup & Restore

Kompleksitas Tinggi
Prioritas Sedang
Apa Ini?

Fitur bagi admin untuk membuat backup (cadangan) database dan file dari dalam admin panel, serta me-restore ke titik tertentu jika terjadi masalah. Ini memberikan admin kontrol penuh atas keamanan data tanpa harus bergantung pada developer atau DBA untuk operasi backup manual.

Jenis Backup
JenisApa yang Di-backupUkuranWaktu
Database FullSeluruh database (tabel, data, stored procedure)Besar5-15 menit
Database DifferentialHanya perubahan sejak full backup terakhirKecil1-3 menit
File/MediaUploaded files (KKJ, CPDP, sertifikat, foto)VariabelTergantung jumlah file
Full SystemDatabase + semua file uploadBesar10-30 menit
Fungsi Utama
  • Manual Backup — Admin klik "Backup Now" kapan saja, misal sebelum update besar
  • Scheduled Backup — Otomatis backup harian/mingguan (cron job)
  • Download Backup — Admin bisa download file backup ke komputer lokal
  • Restore — Pilih backup point → restore database ke kondisi saat itu
  • Retention Policy — Simpan 7 daily + 4 weekly + 3 monthly (rotasi otomatis)
  • Backup History — Daftar semua backup: tanggal, ukuran, tipe, status
Dampak & Manfaat
Disaster Recovery
Jika terjadi kerusakan data, bisa restore dalam hitungan menit — bukan jam atau hari.
Ketenangan Pikiran
Admin tahu data selalu aman karena backup rutin. Bisa berani lakukan perubahan besar.
Compliance
Banyak standar keamanan (ISO 27001, dll) mewajibkan backup rutin + kemampuan restore.
Storage Cost
Backup membutuhkan storage tambahan. Perlu retention policy agar tidak membengkak.
Contoh Skenario di PortalHC
Skenario 1: Sebelum migrasi data besar

Admin mau import 500 data pekerja baru. Sebelumnya, klik "Backup Now". Jika import bermasalah, tinggal restore ke titik sebelum import.

Skenario 2: Data corrupt

Bug di aplikasi menyebabkan data assessment ter-overwrite. Admin buka Backup Manager → pilih backup 1 jam lalu → Restore → data kembali normal.

Skenario 3: Audit reguler

Setiap hari Senin pagi, admin cek Backup History → verifikasi backup otomatis berjalan → download backup mingguan untuk disimpan di drive eksternal.

Peringatan & Risiko
  • Restore = menimpa data saat ini. Data yang dibuat setelah backup point akan hilang. Perlu confirmation berlapis.
  • Backup bisa gagal diam-diam. Perlu monitoring & alert jika scheduled backup gagal.
  • File backup berisi semua data sensitif. Harus dienkripsi dan akses dibatasi.
  • Restore di production server berisiko tinggi. Idealnya test restore di staging dulu.
Implementasi Teknis
  • SQL Server: BACKUP DATABASE / RESTORE DATABASE via SqlCommand
  • Simpan backup di folder terpisah (atau cloud storage: Azure Blob, S3)
  • Hosted Service / Hangfire untuk scheduled backup
  • Halaman admin: Backup Manager (list, create, download, restore, delete)
  • Email/notification jika backup gagal
  • Enkripsi file backup (AES-256)
Platform Referensi
WordPress (UpdraftPlus, BackWPup) cPanel (Backup Wizard) phpMyAdmin (Export/Import) Azure SQL (Automated Backups) AWS RDS (Snapshots)

Ringkasan Rekomendasi Implementasi

🏆 Fase 1 — Quick Wins

Maintenance Mode (1-2 hari) + Dashboard Statistik (3-5 hari). Dampak tinggi, effort rendah.

📢 Fase 2 — Komunikasi

Announcement (3-5 hari) + In-App Notification (5-7 hari). Mengubah cara komunikasi admin-user.

🛡️ Fase 3 — Keamanan Data

Soft Delete (3-5 hari) + Backup & Restore (5-7 hari). Jaring pengaman untuk semua data.

⚙️ Fase 4 — Konfigurabilitas

System Settings (3-5 hari) + User Impersonation (5-10 hari). Admin self-service penuh.

Total estimasi: 24-46 hari kerja untuk ke-8 fitur. Bisa diimplementasi bertahap per fase.
