1. Ringkasan Sistem

Portal HC KPB adalah web portal internal Pertamina untuk Human Capital yang mengelola dua platform utama:

  • CMP (Competency Management Platform) — Assessment online, skills tracking, competency matrix
  • CDP (Competency Development Platform) — IDP, coaching Proton, development plans

Tech Stack: ASP.NET Core MVC 8.0 • SQL Server • Entity Framework Core • SignalR (real-time) • QuestPDF (sertifikat) • ClosedXML (Excel) • Chart.js • Bootstrap 5

Scope Audit: Seluruh sistem Assessment (ujian, soal, scoring, sertifikat) dan Training (records, import, sertifikat, renewal) — tidak termasuk Proton/Coaching.

4. Hasil Analisa — 12 Temuan

KRITIS #1 — ElemenTeknis Score Tidak Dipersist

Lokasi: CMPController.cs (SubmitExam) + AdminController.cs (GradeFromSavedAnswers)

Masalah: Skor per-ElemenTeknis dihitung saat grading (correct/total per ET group) tapi hanya dikirim ke Results view sebagai ViewBag. Tidak disimpan ke database.

Dampak: Tidak bisa jawab "Rata-rata score ET 'Pengetahuan Proses' section X tahun ini?" tanpa recompute dari raw jawaban. Data analytics ET hilang setiap session.

Fix: Tambah tabel SessionElemenTeknisScore (SessionId, ElemenTeknis, CorrectCount, TotalCount). Isi saat SubmitExam/GradeFromSavedAnswers. Effort: LOW.

KRITIS #2 — Tidak Ada Analytics Dashboard HC

Lokasi: Tidak ada halaman yang aggregate data assessment/training secara visual.

Masalah: HC harus export Excel manual setiap kali butuh insight. Tidak ada visualisasi: fail rate per section, per category, trend waktu, sertifikat yang akan expired.

Dampak: Decision-making lambat. HC tidak punya early warning system.

Fix: 1 halaman baru AnalyticsDashboard dengan Chart.js (sudah ada di CDN). Query aggregasi dari data yang sudah ada. Effort: MEDIUM.

KRITIS #3 — Training Compliance Matrix Tidak Ada

Lokasi: WorkerTrainingStatus.CompletionPercentage

Masalah: CompletionPercentage = CompletedTrainings / TotalTrainings di DB. Denominator salah — "total training di DB" ≠ "total training wajib per jabatan". Senior Operator wajib 8 sertifikasi, punya 2 → sistem bilang 100% (2/2) padahal 25% (2/8).

Dampak: Compliance percentage misleading. Regulasi HSE bisa dilanggar tanpa terdeteksi.

Fix: Tabel RequiredTraining (PositionTitle × SubKategori). Admin input matriks training wajib. Query compliance dari sini. Effort: MEDIUM.

SEDANG #4 — Question Bank Tidak Ada

Lokasi: PackageQuestion.AssessmentPackageId → soal terikat per-session

Masalah: Harus import Excel ulang setiap buat assessment baru. Tidak bisa reuse soal. Tidak ada item analysis (difficulty index, discrimination index).

Dampak: Duplikasi effort HC. Tidak bisa improve kualitas soal berdasarkan data historis.

Fix: Model QuestionBank → copy-by-value ke PackageQuestion saat assignment. Exam engine tidak perlu diubah. Effort: HIGH (tapi aman karena copy-by-value).

SEDANG #5 — Tidak Ada Email Notification Sertifikat Expired

Lokasi: Tidak ada background service atau email system.

Masalah: Sertifikat expired silent — hanya terlihat jika HC buka RenewalCertificate page. Tidak ada reminder 90/30/7 hari sebelum expired.

Dampak: HSE compliance risk. Sertifikat keselamatan bisa expired tanpa ada yang tahu.

Fix: IHostedService (BackgroundService) + MailKit NuGet. Tabel NotificationSentLog untuk de-duplikasi. Effort: MEDIUM.

SEDANG #6 — Tab-Switch Detection Tidak Ada

Lokasi: Views/CMP/StartExam.cshtml (JavaScript exam page)

Masalah: ExamActivityLog mencatat: started, page_nav, disconnected, reconnected, submitted — tapi TIDAK ada event untuk visibilitychange / window blur. HC tidak bisa detect apakah worker buka tab lain selama ujian.

Dampak: Integritas ujian berkurang. Worker bisa browsing jawaban di tab lain.

Fix: Tambah visibilitychange event listener → log "focus_lost"/"focus_returned" ke ExamActivityLog via SignalR → tampilkan di MonitoringDetail. JANGAN auto-terminate — HC yang decide. Effort: LOW.

SEDANG #7 — TrainingRecord.Status Lifecycle Ambigu

Lokasi: TrainingRecord.Status — values: "Passed", "Wait Certificate", "Valid"

Masalah: Tidak ada state machine yang jelas. "Passed" dan "Valid" overlap artinya. Expired ditentukan dari ValidUntil bukan Status. Status tidak otomatis berubah saat sertifikat expired.

Dampak: Confusion saat filter/report. HC harus pahami logic gabungan Status + ValidUntil + CertificateType.

Fix: Dokumentasi transisi status yang jelas. Atau simplify: hapus "Valid" (gunakan "Passed" + ValidUntil untuk determine aktif/expired). Effort: LOW.

RENDAH #8 — UserResponse Tanpa Timestamp

Lokasi: Models/UserResponse.cs — legacy path

Masalah: Legacy path jawaban tidak punya field timestamp. PackageUserResponse punya SubmittedAt. Inkonsistensi antara 2 path.

Dampak: Tidak bisa audit waktu per-jawaban di legacy path. Minor karena legacy path jarang dipakai.

Fix: Tambah SubmittedAt ke UserResponse via migrasi. Effort: LOW.

RENDAH #9 — Legacy Question Path Masih Hidup

Lokasi: AssessmentQuestion + AssessmentOption + UserResponse + SaveLegacyAnswer

Masalah: 2 path soal (legacy + package) hidup berdampingan. Menambah complexity di grading logic, views, dan maintenance.

Dampak: Setiap fitur baru harus handle 2 path. Bug potential meningkat.

Fix: Deprecate legacy path. Migrasi existing legacy sessions ke package format. Effort: MEDIUM (perlu migrasi data).

RENDAH #10 — Competency Auto-Update Disabled (Orphaned Tables)

Lokasi: AssessmentCompetencyMap + UserCompetencyLevel — disabled sejak Phase 90

Masalah: Tabel masih ada di DB tapi tidak digunakan. FK ke KkjMatrices (sudah di-drop) orphaned. Dead code di codebase.

Dampak: Database bloat minor. Confusion bagi developer baru.

Fix: Cleanup: drop tabel jika tidak akan dipakai, atau re-enable dengan model baru (OrganizationUnit-based). Effort: LOW (drop) / HIGH (re-enable).

RENDAH #11 — NomorSertifikat Pre-Generated

Lokasi: AdminController.CreateAssessment (POST)

Masalah: NomorSertifikat di-generate saat CreateAssessment, bukan saat ujian selesai dan lulus. Worker yang tidak ikut atau gagal ujian tetap "memakan" nomor sertifikat.

Dampak: Nomor sertifikat tidak sequential (ada gap). Minor karena nomor tetap unique.

Fix: Generate NomorSertifikat saat SubmitExam + IsPassed. Effort: MEDIUM (perlu refactor flow + handle concurrent).

RENDAH #12 — Shared AccessToken

Lokasi: AssessmentSession.AccessToken — shared oleh semua peserta dalam batch

Masalah: Satu token 6 karakter dipakai semua peserta. Jika bocor ke satu orang, semua bisa akses.

Dampak: Risiko rendah karena token hanya untuk memulai, bukan menggantikan autentikasi. HC bisa regenerate.

Fix: Opsional: per-user token. Atau keep as-is (ini pattern yang umum di exam room setting). Effort: LOW.
