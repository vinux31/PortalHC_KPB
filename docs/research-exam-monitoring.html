Overview

Dokumen ini merupakan riset best practices untuk real-time exam monitoring — bagaimana platform terkemuka memberikan visibility kepada admin/pengawas terhadap seluruh peserta yang sedang mengerjakan ujian secara bersamaan.

Konteks Portal HC KPB

Portal KPB sudah memiliki halaman Assessment Monitoring yang mencakup fitur-fitur dasar: status tracking per peserta, token management, Force Close individual, dan Bulk Close. Ini adalah fondasi yang baik dibanding banyak platform kompetitor.


Riset ini fokus mengidentifikasi gap spesifik yang tersisa — terutama live progress per-peserta (soal ke berapa / sudah jawab berapa) dan visual urgency system untuk membedakan peserta yang aktif mengerjakan, idle, atau stuck.

SpeedExam

Model Platform

SpeedExam adalah platform exam online komersial yang fokus pada kemudahan setup dan monitoring real-time. Digunakan untuk corporate assessment, sertifikasi, dan institusi pendidikan. Confidence: MEDIUM-HIGH (dari SpeedExam official features page).

Monitoring Dashboard

SpeedExam menyediakan live monitoring dashboard yang menampilkan semua peserta aktif dalam satu layar:

Aktif — Sedang mengerjakan Belum mulai Selesai (Submitted) Dropped / Koneksi terputus

Deskripsi UI Dashboard SpeedExam

  • List view per peserta: nama, status badge berwarna, browser/OS/IP/lokasi, jumlah attempt
  • Color coding sistematis:
    • Hijau: kandidat sedang aktif mengerjakan
    • Abu-abu: belum mulai atau belum buka link
    • Biru: sudah selesai dan submit
    • Merah: dropped (koneksi terputus atau force-closed)
  • Filter by status: admin bisa filter hanya melihat "Aktif" atau "Dropped" tanpa scroll panjang
  • Per-candidate detail (drill-down): klik nama kandidat → panel expand menampilkan progress detail: soal ke berapa, waktu tersisa, soal yang sudah dijawab
  • Bulk actions toolbar: tombol "Force Close Selected", "Extend Time for Selected" tersedia di atas tabel — tidak perlu keluar dari halaman monitoring
  • Attempt count: jika peserta sudah pernah attempt sebelumnya, jumlah ditampilkan sebagai badge

Fitur Unggulan SpeedExam

Color-Coded Status List

Admin bisa scan status 50+ peserta dalam hitungan detik hanya dari warna baris — tidak perlu baca teks status satu per satu.

Gap: Portal KPB belum ada color urgency pada baris peserta di monitoring

Per-Candidate Progress Drill-Down

Klik satu peserta → panel detail menampilkan "answered X of Y questions, Z minutes remaining". Admin bisa identifikasi peserta yang stuck tanpa keluar dari halaman monitoring.

Gap: Portal KPB (AMON-02) belum ada live progress per peserta

No-Navigation Bulk Actions

Semua action (force close, extend time) bisa dilakukan langsung dari monitoring view — tidak perlu navigasi ke halaman lain. Sangat penting saat monitoring berlangsung.

Portal KPB sudah menerapkan no-navigation principle sebagian

Device Info per Peserta

Browser, OS, IP address, dan lokasi geolokasi tersedia per peserta. Berguna untuk diagnosa jika ada masalah teknis.

Portal KPB: perlu dicek apakah device info tersimpan

Exam.net

Model Platform

Exam.net adalah platform ujian online yang populer di kalangan institusi pendidikan Eropa (terutama Skandinavia). Fokus pada kemudahan invigilator, bukan enterprise proctoring berat. Confidence: MEDIUM (dari web search synthesis).

Deskripsi UI Dashboard Exam.net

Exam.net menggunakan pendekatan minimalis yang mengutamakan kejelasan:

  • Daftar siswa dengan status badge: nama + status real-time (Not Started / Active / Submitted) dalam format list bersih
  • Warna status sederhana: putih (belum mulai), kuning (aktif mengerjakan), hijau (submitted)
  • Hand-raise indicator: ikon tangan terangkat muncul di samping nama siswa jika mereka minta bantuan — invigilator langsung tahu siapa perlu dibantu
  • Chat sidebar per student: invigilator bisa chat privat dengan satu siswa tanpa mengganggu siswa lain
  • Timer override per student: tombol kecil di setiap row untuk tambah waktu ke siswa tertentu

UX Patterns Unggulan Exam.net

Hand-Raise Feature

Peserta bisa "angkat tangan" digital jika ada pertanyaan atau mengalami masalah teknis. Pengawas mendapat notifikasi visual langsung di monitoring dashboard tanpa peserta perlu teriak atau berdiri.

Gap: Portal KPB belum ada fitur hand-raise (nice-to-have untuk future)

Per-Student Chat

Komunikasi langsung antara invigilator dan satu peserta tanpa mengganggu peserta lain. Berguna saat ada klarifikasi soal atau masalah teknis individual.

Gap: Portal KPB belum ada fitur chat (nice-to-have)

Individual Timer Extension

Admin bisa tambah waktu ke satu peserta tertentu (misalnya karena gangguan koneksi) tanpa mengubah waktu untuk semua peserta.

Gap: Portal KPB timer adalah global per session, belum bisa per peserta

Simplisitas Dashboard

Tampilan sangat bersih — hanya informasi yang kritis yang ditampilkan. Tidak ada clutter. Invigilator dapat fokus pada monitoring, bukan memahami UI.

Portal KPB bisa adopsi pendekatan progressive disclosure ini

Online Proctoring Patterns 2024-2025

Berdasarkan sintesis dari beberapa sumber (SpeedExam documentation, Smashing Magazine 2025 UX strategies, industri-wide review), berikut adalah pola-pola yang muncul berulang di platform monitoring terbaik.

Pattern 1: Progressive Disclosure

Prinsip: Informasi kritis (nama, status, waktu sisa) ditampilkan langsung di daftar. Detail tambahan (soal ke berapa, device info, history attempt) baru muncul saat admin klik/expand row tersebut.

Alasan efektif: Admin yang memantau 50+ peserta tidak bisa membaca semua detail sekaligus. Progressive disclosure memastikan informasi paling penting selalu terlihat, dan detail hanya dimuat saat dibutuhkan.

Pola ini bisa diadopsi Portal KPB: tampilkan answered/total di kolom utama, detail lain di drill-down

Pattern 2: No-Navigation Principle

Prinsip: Semua action yang mungkin dibutuhkan selama monitoring (force close, extend time, view detail) harus bisa dilakukan dari halaman monitoring itu sendiri — tanpa navigasi ke halaman lain.

Alasan kritis: Jika admin harus navigasi ke halaman lain untuk melakukan action, mereka kehilangan visibility terhadap peserta lain selama transisi. Dalam konteks ujian real-time, setiap detik kehilangan visibility adalah risiko.

Portal KPB sudah menerapkan ini sebagian — Force Close dan Bulk Close ada di halaman monitoring

Pattern 3: Color Urgency System

Prinsip: Gunakan warna untuk menyampaikan status tanpa membutuhkan pembacaan teks.

Hijau: Aktif mengerjakan Kuning: Idle / Potensi masalah Merah: Dropped / Expired / Force Closed Abu: Belum mulai Biru: Selesai (Submitted)

Alasan efektif: Manusia memproses warna jauh lebih cepat dari teks. Admin yang melihat 3 baris merah sekaligus langsung tahu ada masalah tanpa membaca satu per satu.

Gap: Portal KPB belum menggunakan color urgency pada daftar peserta di monitoring

Pattern 4: Alert / Flag System

Prinsip: Perilaku abnormal (idle terlalu lama, tab switching terdeteksi, koneksi terputus berulang) langsung ditandai dengan flag atau ikon pada baris peserta — tanpa admin perlu mengamati terus-menerus.

Implementasi: Badge merah kecil di sudut baris, atau baris langsung berubah warna, saat flag terdeteksi. Admin bisa klik untuk melihat detail pelanggaran.

Terkait AINT-02/03 (tab-switch detection) yang sedang dikembangkan di v8.0 — integration dengan monitoring view adalah langkah selanjutnya

Pattern 5: Bulk Actions dengan Seleksi

Prinsip: Checkbox per row memungkinkan admin memilih subset peserta dan menerapkan action sekaligus (misalnya: close semua yang sudah idle 15 menit).

Kombinasi dengan filter: Paling efektif ketika dikombinasikan dengan filter status — filter "Dropped", centang semua, klik "Archive". Satu langkah untuk bersihkan data kotor.

Portal KPB sudah punya Bulk Close (AMON-03) — fondasi sudah ada
