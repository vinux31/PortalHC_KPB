## 1. Background & Motivation

Phase 323 (commit range `392f0b24..f1849367`) fixed cascade FK bugs in 3 endpoints under `Controllers/AssessmentAdminController.cs` (Manage Assessment Groups tab — `DeleteAssessment`, `DeleteUPA`, and related). Scope was deliberately tight per `323-CONTEXT.md:122` + `323-RESEARCH.md:43`, deferring remaining delete endpoints to a future "v19.0 Cascade Audit Sweep" milestone slot.

Pre-audit analysis on 2026-05-27 (this conversation) confirmed that `DeleteTraining` (`Controllers/TrainingAdminController.cs:527-548`) and `DeleteManualAssessment` (`Controllers/TrainingAdminController.cs:736-756`) — both exposed via `Views/Admin/Shared/_TrainingRecordsTab.cshtml:327-349` (Input Records tab) — exhibit at least 3 categories of edge-case failure:

1. **Renewal chain FK violation** — `TrainingRecord.RenewsTrainingId` and `AssessmentSession.RenewsTrainingId`/`RenewsSessionId` use `OnDelete(NoAction)` (`Data/ApplicationDbContext.cs:157-165, 220-228`). App code does not null-clear referrer rows before `Remove`, so SQL Server raises FK violation → 500 error if any child row references the target as renewal source.
2. **File–DB atomicity broken** — both endpoints delete the certificate file from `wwwroot/uploads/...` *before* `SaveChangesAsync`. If DB save fails (FK violation above, or `AssessmentEditLog`/`PackageUserResponse` Restrict), the file is gone but the row remains → orphan record + permanent certificate loss.
3. **FK Restrict on dependent tables** — `AssessmentEditLog → AssessmentSession` (`OnDelete(Restrict)` per `Data/ApplicationDbContext.cs:236-246`, added in Phase 321) and `PackageUserResponse → AssessmentSession` (Restrict, `:485-490`) will block delete if any rows exist. Probability is low for manual-entry sessions but non-zero — schema allows it.

The bug surface is wider than these 2 endpoints. A systematic audit is the right next step *before* committing to fix work, because the fix pattern depends on per-endpoint cascade graph shape (renewal chain only applies to training/assessment, others have different FK risks). Auditing first prevents scope creep in fix phases and produces a prioritized backlog for v19.0/v20.0.

## 2. Goal

Produce a comprehensive, evidence-backed audit report (`328-RESEARCH.md`) enumerating every `Delete*` method in `Controllers/*.cs` and `Services/*.cs`, evaluating each against a 7-dimension cascade-safety checklist, assigning a severity tag, and listing remediation outlines — **without writing any fix code**. The audit becomes the input artifact for subsequent fix-phase planning decisions.

## 3. Non-Goals

This phase **does NOT**:

- Modify any controller, service, model, view, migration, or test code.
- Spawn fix phases automatically.
- Produce stakeholder-facing HTML reports or IT_NOTIFY documents.
- Audit `UserManager.DeleteAsync` (Identity framework, separate concern).
- Evaluate soft-delete / `IsDeleted` flag adoption (product decision, not an audit concern).
- Evaluate concurrency / optimistic locking patterns (v20.0+ concern).
- Evaluate idempotency / re-delete behavior.
- Audit cascade behavior in stored procedures (none confirmed in codebase).

## 4. Audit Methodology

### 4.1 Endpoint Enumeration

Target regex: `public async.*Delete\w+\(` matched against:

- `Controllers/*.cs` (all controller files)
- `Services/*.cs` (all service files containing `_context.Remove` / `RemoveRange` calls)

Estimated row count: 15–25 endpoints (pre-audit estimate; final count is part of the deliverable).

Each match becomes one row in the audit table.

### 4.2 Seven-Dimension Checklist

| # | Dimension | What to check | Pass criteria |
|---|-----------|---------------|---------------|
| 1 | Cascade FK risk | EF Core `OnDelete(Restrict\|NoAction)` in `Data/ApplicationDbContext.cs` for all FKs pointing to the target entity | ✅ All Restrict/NoAction FKs handled at app level (null-clear or pre-delete of dependents) |
| 2 | File–DB atomicity | Order of file delete vs `SaveChangesAsync` | ✅ DB save executes before file delete; OR transaction wrap with try/catch rollback |
| 3 | AuditLog presence | `_auditLog.LogAsync(...)` call with action `"Delete"` | ✅ Present with entity name + ID captured |
| 4 | Role/auth check | `[Authorize(Roles = "...")]` attribute | ✅ Attribute present with appropriate role (Admin / HC / specific) |
| 5 | Renewal chain null-clear | For `TrainingRecord` / `AssessmentSession`: child rows referring to target via `RenewsTrainingId` / `RenewsSessionId` must be null-cleared before `Remove` | ✅ Pre-delete: `UPDATE child SET RenewsXId = NULL WHERE RenewsXId = target.Id` executed |
| 6 | Error handling | `try/catch (DbUpdateException)` with friendly `TempData["Error"]` | ✅ FK violation caught, friendly error returned, no 500 thrown |
| 7 | Transaction wrap | `await using var tx = _context.Database.BeginTransactionAsync()` for multi-step deletes (file + DB + cascade pre-clear) | ✅ Transaction present for multi-step; single-table delete may skip transaction |

Per-row evidence column: `file.cs:line` reference for each ✅/❌/⚠️ verdict.

### 4.3 Severity Rubric

- **HIGH** — Dim 1, 2, or 5 fail. Justification: data loss possible (FK violation throws 500 to user; orphan file or row left in DB).
- **MED** — Dim 3, 4, or 6 fail. Justification: UX degradation (generic 500 instead of friendly error; missing audit trail; potentially missing role check).
- **LOW** — Dim 7 fail. Justification: multi-step delete without transaction wrap is a hygiene issue but rarely causes user-visible failure on modern SQL Server.

### 4.4 Reference Gold Standard

Compare each endpoint's pattern against `Controllers/AssessmentAdminController.DeleteAssessment` post-commit `f1849367` (Phase 323 final). Capture the canonical remediation snippet at the bottom of `328-RESEARCH.md` under a `Remediation Pattern Template` section, for use by future fix-phase planners.
