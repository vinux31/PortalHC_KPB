2. Mode Autentikasi

Sistem mendukung 3 mode autentikasi yang dapat dikonfigurasi via appsettings.json. Pemilihan mode dilakukan di Program.cs saat aplikasi start.

Development

LocalAuthService

Autentikasi menggunakan password hash di database lokal (ASP.NET Core Identity). Digunakan saat development.

  • Password di-hash via Identity
  • Tanpa lockout (aplikasi intranet)
  • Config: UseActiveDirectory = false
Production

LdapAuthService

Autentikasi langsung ke Active Directory Pertamina via LDAP. Validasi kredensial dilakukan oleh AD server.

  • LDAP bind ke OU=KPB,OU=KPI,DC=pertamina,DC=com
  • Timeout 5 detik (cegah hang)
  • Sync atribut: displayName, mail
  • Proteksi LDAP injection (RFC 4515)
Production (Aktif)

HybridAuthService

Mode utama di production. AD-first dengan fallback ke local khusus akun admin.

  • User biasa: AD only
  • admin@pertamina.com: AD dulu, gagal → fallback local
  • Fallback silent (tidak ada indikasi ke user)
  • Config: UseActiveDirectory = true
⚠
Penting: User tidak otomatis ter-create saat login via AD. HC harus mendaftarkan user terlebih dahulu di sistem (pre-registration). AD hanya memvalidasi kredensial.

Konfigurasi Autentikasi

// appsettings.json { "Authentication": { "UseActiveDirectory": true, "LdapPath": "LDAP://OU=KPB,OU=KPI,DC=pertamina,DC=com", "LdapTimeout": 5000, "AttributeMapping": { "Email": "mail", "FullName": "displayName", "NIP": "employeeID" } } }

4. Sistem Role & Hirarki

Sistem menggunakan 6 level hirarki role. Setiap role menentukan akses fitur dan tampilan default portal.

Level Role Tampilan Default Akses
1 Admin Admin Full access — semua fitur & data
2 HC HC Full access — kelola pekerja, data, laporan
3 Direktur VP Manager Atasan Full access — lihat semua data lintas unit
4 Section Head Sr. Supervisor Atasan Section access — data di bagian sendiri
5 Coach Supervisor Coach Coaching access — kelola coachee yang di-assign
6 Coachee Coachee Self-access — data pribadi saja

Helper Methods (UserRoles.cs)

MethodLogika
HasFullAccess(level)Level ≤ 3 (Admin, HC, Direktur, VP, Manager)
HasSectionAccess(level)Level = 4 (Section Head, Sr. Supervisor)
IsCoachingRole(level)Level ≥ 5 (Coach, Supervisor, Coachee)
GetDefaultView(role)Mapping role → tampilan portal default
🛈
Otorisasi di Controller: Menggunakan [Authorize(Roles = "Admin, HC")] pada action yang membutuhkan role tertentu. Tidak menggunakan custom policy — murni role-based.

5. Registrasi & Manajemen User

Tidak ada registrasi mandiri. Semua user harus didaftarkan oleh Admin atau HC melalui menu Kelola Data.

Langkah 1

HC akses Kelola Data → Manajemen Pekerja

Halaman daftar semua pekerja dengan filter (bagian, unit, role, status aktif).

Langkah 2A — Satu per satu

Tambah Pekerja (CreateWorker)

Isi form: Nama, Email, NIP, Jabatan, Bagian, Unit, Direktorat, Role, Tanggal Bergabung. Di mode lokal, juga isi password.

Langkah 2B — Massal

Import via Excel (ImportWorkers)

Download template Excel → isi data → upload. Sistem memproses baris per baris dengan validasi otomatis. Hasil ditampilkan: sukses, error, skip, atau perlu review.

Pasca Registrasi

User bisa login

Di mode AD: user login dengan kredensial Active Directory Pertamina. Di mode lokal: login dengan password yang diset saat pembuatan akun.

Aktivasi & Deaktivasi

HC dapat menonaktifkan akun pekerja tanpa menghapus data (soft-delete). User yang di-nonaktifkan tidak bisa login dan akan melihat pesan error. HC juga bisa mengaktifkan kembali akun kapan saja.

ReactivateWorker
IsActive = true → user bisa login lagi
DeactivateWorker
IsActive = false → user tidak bisa login

Seed Data (Bootstrap)

Saat pertama kali dijalankan, sistem otomatis membuat:

  • Semua role di database (Admin, HC, Direktur, VP, Manager, dll.)
  • Akun bootstrap: admin@pertamina.com / 123456 (harus ganti password)
  • Development only: 8 user test dengan berbagai role untuk testing
  • Data OrganizationUnits (4 bagian, 15+ unit)

6. Sesi & Cookie

Autentikasi menggunakan cookie-based session dari ASP.NET Core Identity.

Konfigurasi Cookie

  • Login Path /Account/Login
  • Logout Path /Account/Logout
  • Access Denied Path /Account/AccessDenied
  • Session Timeout 8 jam (idle)
  • Sliding Expiration Aktif — timeout di-reset setiap request
  • SameSite Policy Lax — cegah CSRF cross-origin
  • Secure Policy SameAsRequest (HTTPS di production)
  • Remember Me Opsional — cookie persisten jika dicentang

SignalR Hub (WebSocket)

Endpoint /hubs/* mendapat perlakuan khusus: jika cookie expired, server mengembalikan HTTP 401 (bukan redirect ke halaman login). Ini mencegah WebSocket connection rusak karena menerima HTML redirect response.

Proses Logout

User klik Logout → POST /Account/Logout → SignOutAsync() menghapus cookie → redirect ke halaman Login.

7. Keamanan

✓
AntiForgeryToken — Semua form POST dilindungi CSRF token
✓
Open Redirect Prevention — returnUrl divalidasi via Url.IsLocalUrl()
✓
LDAP Injection Protection — Input di-escape sesuai RFC 4515
✓
Generic Error Messages — Pesan error tidak mengekspos detail teknis
✓
Password Hashing — Identity hash (PBKDF2) untuk mode lokal
✓
LDAP Timeout — 5 detik timeout mencegah resource exhaustion
✓
SameSite Cookie — Policy Lax melindungi dari CSRF cross-site
✓
Pre-Registration Required — User harus didaftarkan HC, tidak bisa self-register
⚠
Catatan: Sistem tidak menerapkan account lockout karena berjalan di jaringan intranet di belakang firewall. Jika diperlukan, lockout bisa diaktifkan di konfigurasi Identity.
